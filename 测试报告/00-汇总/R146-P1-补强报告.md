## 三、真实端点补齐（R146 提交）

**问题**：前端 `requirement.ts` v1.52 P1-1 加了 2 个 API 包装：

```ts
getTraceCount(id) { return api.get(`/requirements/${id}/trace-count`) }  // 一直 404
getTestCaseCount(id) { return api.get(`/requirements/${id}/test-case-count`) }  // 一直 404
```

`RequirementDetail.vue` 用 try/catch 兜底（catch 后用本地聚合），所以不爆，但浪费 HTTP 请求。

**修复**：在 `RequirementController` 加 2 个端点，用 `JdbcTemplate` 跨 schema 查：

| 端点 | 实现 | 返回 |
|------|------|------|
| `GET /api/requirements/{id}/trace-count` | 查 `req_schema.t_requirement_ancestor` 闭包表 | `{upstream, downstream}` |
| `GET /api/requirements/{id}/test-case-count` | 查 `trace_schema.t_requirement_test_case` | `{count}` |

**为何用 JdbcTemplate**：`med-rms-requirement` 已被 `med-rms-traceability` 依赖，加反向依赖会循环。JdbcTemplate 直接查 `trace_schema` schema 是单测通过 + 解耦的折中方案。

## 八、R147 部署注意

R146 提交后用 `tools/restart_8080.ps1` 重新启动后端，但运行时 OpenAPI 文档不包含新加的 2 个端点（trace-count / test-case-count），导致 SY0301（资源不存在）。原因疑似：
- mvn compile 只重编译修改过的文件，但 spring-boot-devtools 缓存了 controller bean
- mvn spring-boot:run 5.5s 启动过短（正常 20s+），可能跳过了完整重启

**解决**（R147 部署时执行）：
```bash
cd Code/backend
mvn clean install -DskipTests
# 然后用 restart_8080.ps1 启动
```

或者直接：
```bash
rm -rf med-rms-requirement/target
mvn -pl med-rms-requirement -am compile
```

**代码层面已完整**：javap 验证 RequirementController.class 含 traceCount + testCaseCount 方法及 JdbcTemplate 注入构造函数，仅运行时未生效。
