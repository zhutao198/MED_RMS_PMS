## 1. 产品目标

本系统服务于武汉中旗生物医疗电子有限公司内部，解决医疗器械需求管理中合规追溯难、变更影响不可控、层级转化无标准三大核心痛点。

| 目标编号 | 目标 | 衡量方式 | 目标值 |
|---------|------|---------|-------|
| G1 | **合规过程受控**：确保需求管理合规过程系统化、可审计、可度量 | 审计发现项闭环率 + 合规性检查通过率 | 闭环率≥95%；检查通过率≥98%；关键/重大缺陷闭环周期≤30天 |
| G2 | **追溯全链路**：实现URS→PRS→SRS→DRS四层需求纵向可追溯，分阶段达标 | 需求全链路闭环率 | MVP≥85%→V1.0≥95%→V2.0≥99% |
| G3 | **变更零失控**：任何需求变更的影响范围可自动评估、受影响项可自动标记、追溯链可自动更新 | 变更影响评估覆盖率 | 100% |

**目标正交性说明**：G1关注合规过程受控，G2关注数据完整性，G3关注过程可控性——三者互不包含、缺一不可。

> **变更记录**：2026-05-22 | 许清楚 | 修改 | P-RED-02：将"合规零缺陷"改为可度量的过程指标组合。原G1"合规零缺陷=审计不符合项=0"不可度量且不符合IEC 62304 Clause 9理念（承认问题会发生，关键是有系统化处理流程）。新增G1采用闭环率+检查通过率双指标，并增加缺陷闭环周期约束。G2追溯覆盖率改为分阶段递进目标，与IEC 62304 Clause 5.2.2对齐。

## 2. 用户故事

### US-1 研发总监：全局审批监督
> 作为研发总监，我需要在仪表盘上一目了然地看到所有需求的状态分布、合规达标率和追溯完整率，以便我在DCP阶段门评审时快速判断项目是否具备进入下一阶段的条件。

**验收标准**：
- **功能验收**：Given 系统中存在处于DCP门的项目 / When 研发总监打开仪表盘 / Then 可看到该项目需求合规率、追溯闭环率、高风险需求数的汇总卡片，且数据实时更新（延迟≤5分钟）
- **合规验收**：仪表盘数据需满足21 CFR Part 11 11.10(e)审计追踪要求，所有指标计算过程可审计（对应ISO 13485 Clause 7.3.7设计评审记录）
- **性能验收**：项目需求量≤5000条时，仪表盘加载时间≤3秒

### US-2 项目经理：变更影响全景掌控
> 作为项目经理，我需要当任何需求发生变更时，系统自动评估并展示所有受影响的下游需求、设计文档和测试用例，以便我准确评估变更代价并组织评审。

**验收标准**：
- **功能验收**：Given URS-001发生变更 / When 项目经理打开变更影响评估面板 / Then 系统自动列出所有下游PRS/SRS/DRS及关联测试用例，标记为"suspect"状态，并计算影响范围百分比
- **合规验收**：变更影响评估过程须完整记录审计日志（对应21 CFR Part 11 11.10(e)），影响评估报告可作为DHF证据包的一部分（对应IEC 62304 Clause 8.2.2变更控制记录）
- **性能验收**：需求量≤5000条时，影响评估生成时间≤10秒

### US-3 系统架构师：L2→L3高效转化
> 作为系统架构师，我需要在可视化拆解工作台中，将一个PRS逐条拆解为多个SRS，并自动继承上游的法规关联和追溯关系，以便我高效完成需求层级转化而不遗漏合规约束。

**验收标准**：
- **功能验收**：Given PRS-010已通过评审 / When 系统架构师在拆解工作台中创建SRS / Then 系统自动继承PRS-010的法规条款关联，并在拆解完成后校验所有SRS是否覆盖PRS的全部验收标准
- **合规验收**：拆解转化操作需触发电子签名（对应21 CFR Part 11 11.50签署显现），继承的法规条款关联不得遗漏（对应IEC 62304 Clause 5.2.2可追溯性要求）
- **性能验收**：单次拆解操作（1条→≤10条下游）响应时间≤5秒

### US-4 合规专员：合规证据一键生成
> 作为合规专员，我需要在项目DCP评审前，一键生成包含完整追溯矩阵、电子签名记录和审计日志的DHF合规证据包，以便我高效应对审计而不用手工拼凑文档。

**验收标准**：
- **功能验收**：Given 项目完成某阶段所有需求评审 / When 合规专员点击"生成证据包" / Then 系统在≤30秒内生成包含追溯矩阵、评审记录、电子签名日志的PDF证据包，且追溯闭环率≥95%
- **合规验收**：证据包内容需覆盖NMPA eRPS格式要求（对应NMPA医疗器械软件注册审查指导原则），支持按目标市场选择模板（NMPA/FDA/CE），电子签名记录需满足11.50签署显现要求（签名者+日期+意义不可分离）
- **性能验收**：项目需求量≤2000条时，证据包生成时间≤30秒

### US-5 产品经理：L1/L2需求高效生产
> 作为产品经理，我需要在需求收集池中汇总来自客户、法规、市场的多渠道需求，并将其转化为结构化的URS和PRS，以便我系统化地管理需求来源和优先级。

**验收标准**：
- **功能验收**：Given 多条需求进入收集池 / When 产品经理选择并转化为URS / Then 系统自动记录需求来源、原始描述，并引导填写URS必填字段（法规关联、优先级、验收标准）
- **合规验收**：URS创建操作需触发电子签名（对应21 CFR Part 11 11.70签名与记录不可分离链接），需求来源追溯记录满足ISO 13485 Clause 7.3.3设计输入要求
- **性能验收**：单条URS创建+保存响应时间≤3秒

### US-6 研发工程师：L4需求落地执行
> 作为研发工程师，我需要从SRS直接接收拆解后的DRS，并在完成实现后标记对应测试用例为"待验证"，以便需求实现和验证之间的衔接不断裂。

**验收标准**：
- **功能验收**：Given SRS-020已拆解为DRS / When 研发工程师标记DRS-020-01为"已实现" / Then 系统自动关联的测试用例状态更新为"待验证"，并通知测试工程师
- **合规验收**：DRS实现完成标记需记录审计日志（对应IEC 62304 Clause 5.8软件验收测试记录），关联测试用例覆盖状态需满足Clause 5.7追溯性要求
- **性能验收**：DRS状态变更+关联测试用例状态更新操作≤2秒

### US-7 测试工程师：需求-用例覆盖验证
> 作为测试工程师，我需要在创建测试用例时直接关联到SRS/DRS，并实时查看需求覆盖率，以便我确保每个需求都有对应的验证用例且无遗漏。

**验收标准**：
- **功能验收**：Given SRS-020处于"已评审"状态 / When 测试工程师创建测试用例并关联 / Then 系统实时更新SRS-020的测试覆盖率，若覆盖率<100%则在该需求卡片上显示警告标识
- **合规验收**：测试用例与需求的双向追溯关系需满足IEC 62304 Clause 5.7.1软件测试可追溯性要求，覆盖率数据需纳入审计日志
- **性能验收**：测试用例关联操作响应时间≤2秒，覆盖率计算实时更新（延迟≤5秒）

### US-8 质量工程师：全层质量监督
> 作为质量工程师，我需要在任何需求评审未通过或追溯链断裂时收到系统通知，以便我及时介入跟踪整改，确保质量体系持续有效。

- **性能验收**：通知送达延迟≤1分钟（站内信），邮件通知≤5分钟

### 7.1 需求层级架构

#### 7.1.1 四层架构总览

```
┌─────────────────────────────────────────────────────────┐
│ L1: URS（用户需求规格）                                    │
│ 来源：用户/客户/市场/法规                                   │
│ ↓ N:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L2: PRS（产品需求规格）                                    │
│ 来源：URS拆解+产品定义                                     │
│ ↓ 1:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L3: SRS（系统需求规格）                                    │
│ 来源：PRS拆解+系统架构约束                                  │
│ ↓ 1:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L4: DRS（设计需求规格）                                    │
│ 来源：SRS拆解+设计约束                                     │
│ → 关联：测试用例、设计文档、代码模块                          │
└─────────────────────────────────────────────────────────┘
```

#### 7.1.2 L1: URS — 用户需求规格

| 属性 | 定义 |
|------|------|
| **核心定义** | 从用户/客户/法规视角描述的需求，回答"用户需要什么" |
| **边界** | 不涉及技术实现方案，仅描述用户可感知的行为或约束 |
| **主要生产者** | 产品经理 |
| **主要消费者** | 产品经理（评审）、合规专员（合规审查）、系统架构师（下游转化输入） |
| **IPD阶段门映射** | DCP1（概念阶段）输入 |

**必填字段清单**：

| 字段名 | 类型 | 说明 | 校验规则 |
|--------|------|------|---------|
| URS-ID | 自动生成 | 格式：URS-{项目编号}-{序号} | 系统自动，不可修改 |
| 标题 | 文本 | 需求简述，≤50字 | 非空 |
| 详细描述 | 富文本 | 完整需求描述 | 非空，≥20字 |
| 需求来源 | 枚举 | 客户/市场/法规/内部/竞品 | 非空 |
| 来源编号 | 文本 | 原始需求编号/法规条款号 | 非空（来源为法规时必填） |
| 法规关联 | 多选关联 | 关联的法规条款ID列表 | 来源为法规时至少1条 |
| 优先级 | 枚举 | 必须/应该/可选（MoSCoW） | 非空 |
| 验收标准 | 富文本列表 | 可验证的通过条件 | 非空，至少1条 |
| 风险等级 | 枚举 | 高/中/低 | 默认"中" |
| 需求分类 | 枚举 | 硬件/软件/软硬件（新增） | 非空，默认"软件" |
| 所属项目 | 关联 | 项目ID | 非空 |
| 创建人 | 自动 | 当前用户 | 系统自动 |
| 创建时间 | 自动 | 时间戳 | 系统自动 |

> **变更记录**：2026-05-22 | 许清楚 | 修改 | P-ORG-09：URS新增"需求分类"字段（硬件/软件/软硬件），支持URS层记录硬件需求条目（轻量级，仅编号+描述+状态），不提供硬件需求的深度管理（BOM、版本等）。对应NG-9的调整。

**状态机定义**：

```
草稿 → 评审中 → 已通过 → 已基线化 → 已变更 → 评审中（循环）
                ↘ 已驳回 → 草稿（修改后重新提交）
```

#### 7.1.6 层级间转化规则

| 转化路径 | 基数 | 转化条件 | 转化时自动执行 | 校验规则 |
|---------|------|---------|-------------|---------|
| URS→PRS | 1:N | URS状态为"已通过" | 法规关联继承、来源追溯建立 | 每条URS至少拆解为1条PRS；所有PRS验收标准覆盖URS验收标准 |
| PRS→SRS | 1:N | PRS状态为"已通过" | 法规关联继承、接口需求传递 | 每条PRS至少拆解为1条SRS；所有SRS验收标准覆盖PRS验收标准 |
| SRS→DRS | 1:N | SRS状态为"已通过" | 法规关联继承、接口规格传递、测试策略继承 | 每条SRS至少拆解为1条DRS；所有DRS验收标准覆盖SRS验收标准 |

**转化完整性校验**（拆解工作台内置）：
1. **覆盖率校验**：下游所有验收标准的并集是否覆盖上游全部验收标准
2. **法规一致性校验**：下游的法规关联是否包含上游的全部法规关联
3. **完整性校验**：所有下游需求的必填字段是否均已填写

**反向追溯（N:1汇总）**：
- 允许多条URS汇总为1条PRS（当多个用户需求指向同一产品能力时）
- 允许多条PRS汇总为1条SRS（当多个产品需求通过同一系统方案实现时）
- 系统记录每条追溯关系的创建时间、创建人、追溯原因

#### 7.2.1 需求创建与编辑（FR-0.6）

**功能描述**：支持四层需求的创建、编辑，按层级自动校验必填字段。

#### 7.2.2 需求状态机流转（FR-0.1）

**功能描述**：每层需求按7.1节定义的状态机流转，所有状态变更记录在审计日志中。

**验收标准**：
- Given 需求当前状态为S1 / When 用户执行操作O / Then 需求状态变更为S2，审计日志记录变更人、时间、原状态、新状态
- Given 需求当前状态为S1 / When 用户尝试执行不允许的操作O' / Then 系统拒绝操作并提示"当前状态不允许此操作"
- Given 需求已被基线化 / When 发起变更 / Then 状态变为"已变更"，所有下游关联项自动标记为"suspect"（见7.4节）

#### 7.2.3 需求评审在线化（FR-0.4）

**功能描述**：评审发起→评审执行→评审结论全在线，评审意见结构化可追溯。

**评审流程**：

```
发起评审 → 系统通知评审人 → 评审人查看需求 → 提交评审意见
→ 汇总评审结论 → 通过/驳回
```

**评审配置**：
- 评审类型：同行评审、技术评审、合规评审
- 评审人：按需求层级自动推荐（URS→产品经理+合规专员；SRS→系统架构师+合规专员；DRS→系统架构师+研发工程师）
- 评审意见字段：结论（通过/有条件通过/驳回）、意见类型（建议/问题/致命）、详细描述、关联需求片段

**验收标准**：
- Given 需求状态为"草稿" / When 创建人提交评审 / Then 系统通知所有评审人，需求状态变为"评审中"，评审截止时间默认7个工作日
- Given 评审人提交评审意见 / When 所有评审人完成评审 / Then 系统汇总评审结论：无致命问题且通过率>50%→"已通过"；否则→"已驳回"
- Given 评审结论为"有条件通过" / When 创建人确认修改意见 / Then 系统记录修改内容，状态变为"已通过"
- Given 任意评审发生 / When 评审完成 / Then 评审记录（评审人、时间、意见、结论）永久保存，不可删除

#### 7.2.4 基线快照管理（FR-0.8）

**功能描述**：对任意层级的需求集合创建基线快照，基线不可修改但可对比差异。

**验收标准**：
- Given 项目经理选择一组需求 / When 创建基线 / Then 系统生成基线快照（含所有需求的当前状态、字段值、追溯关系），基线编号格式：BL-{项目编号}-{序号}-{日期}
- Given 基线已创建 / When 用户尝试修改基线内容 / Then 系统拒绝操作并提示"基线不可修改"
- Given 基线已创建 / When 用户选择与另一基线对比 / Then 系统展示两个基线之间的需求差异（新增/删除/修改的需求列表，字段级变更详情）
- Given 项目到达DCP阶段门 / When 项目经理创建基线 / Then 系统自动命名为"DCP{N}-基线-{日期}"

#### 7.2.5 拆解工作台（FR-0.7 / FR-1.1）

**功能描述**：核心工作界面，支持需求的逐层拆解转化。

**工作台功能**：
1. **树形视图**：左侧展示当前项目的需求层级树，支持展开/折叠
2. **拆解面板**：右侧展示选中需求的详情和拆解操作区
3. **拆解操作**：选择上游需求→点击"创建下游需求"→自动继承上游关联→填写下游字段→保存

5. **看板视图**（P1增强）：拖拽卡片式拆解，支持并行拆解多条需求

#### 7.3.1 纵向追溯（FR-0.5）

### 7.4 变更管理

> **设计原则**：变更管理不是需求管理的子环节，而是跨模块枢纽功能。变更影响需求、追溯、合规、风险、测试全链路。

#### 7.4.1 变更发起与分类（FR-1.7）

**变更类型**：

| 变更类型 | 触发条件 | 审批级别 |
|---------|---------|---------|
| 纠正性变更 | 需求描述错误/不完整 | 项目经理审批 |
| 适应性变更 | 法规/标准更新 | 合规专员+项目经理审批 |
| 完善性变更 | 功能优化/新需求 | 项目经理+研发总监审批 |
| 紧急变更 | 安全/合规风险 | 研发总监直接审批，事后补流程 |

**变更申请必填字段**：
- 变更原因（枚举+详细描述）
- 变更内容（修改前后对比）
- 影响范围（系统自动填充，人工确认）
- 紧急程度（普通/紧急）
- 期望完成日期

**验收标准**：
- Given 基线化后的需求 / When 用户点击"发起变更" / Then 系统弹出变更申请表单，自动填充需求当前内容
- Given 变更申请已提交 / When 审批人查看 / Then 可看到修改前后对比（diff视图）、自动影响评估报告、审批操作

#### 7.4.2 变更影响自动评估（FR-1.3）

**功能描述**：变更提交时自动生成影响范围报告。

**影响评估维度**：

| 维度 | 评估内容 | 自动/手动 |
|------|---------|----------|
| 下游需求 | 受影响的PRS/SRS/DRS列表 | 自动 |
| 测试用例 | 受影响的测试用例列表 | 自动 |
| 法规影响 | 受影响的法规条款合规性 | 自动 |
| 横向关联 | 存在依赖/冲突关系的同层需求 | 自动 |
| SOUP影响（新增） | 变更涉及的SOUP组件 | 自动 |
| 工期影响 | 预估影响的工作量（人天） | 手动填写 |
| 风险影响 | 变更引入的新风险 | 手动填写 |

> **变更记录**：2026-05-22 | 许清楚 | 修改 | P-ORG-01：变更影响评估新增SOUP影响维度

**验收标准**：
- Given URS-001发起变更 / When 系统执行影响评估 / Then 自动列出所有下游PRS/SRS/DRS、关联测试用例、法规条款、涉及SOUP组件，计算影响范围（受影响项数/总关联项数×100%）
- Given 影响评估完成 / When 项目经理查看报告 / Then 报告包含：受影响需求数、受影响测试用例数、法规影响标记、SOUP影响标记、预估影响范围百分比

#### 7.4.3 下游自动标记suspect（FR-0.10）

**功能描述**：上游需求变更后，所有下游关联项自动标记为"suspect"（源自Jama Connect最佳实践）。

**验收标准**：
- Given URS-001变更已批准 / When 系统处理变更 / Then 所有下游PRS/SRS/DRS自动标记为"suspect"，状态不变但显示黄色警告标识
- Given 下游需求被标记为"suspect" / When 负责人确认无需修改 / Then 手动清除"suspect"标记，系统记录确认人和时间
- Given 下游需求被标记为"suspect" / When 负责人修改需求 / Then 修改保存后"suspect"标记自动清除，系统记录修改内容

#### 7.4.4 变更审批流（FR-1.7）

**审批流程**：

```
变更申请 → 影响评估 → 审批 → 执行 → 验证 → 关闭
```

| 阶段 | 执行者 | 产出 |
|------|--------|------|
| 变更申请 | 需求负责人 | 变更申请单 |
| 影响评估 | 系统+项目经理 | 影响评估报告 |
| 审批 | 按变更类型确定审批人 | 审批结论 |
| 执行 | 需求负责人 | 修改后需求 |
| 验证 | 测试工程师+质量工程师 | 验证结论 |
| 关闭 | 项目经理 | 变更记录归档 |

**验收标准**：
- Given 变更审批通过 / When 需求负责人执行修改 / Then 系统记录修改diff，更新追溯关系，通知所有受影响项的负责人
- Given 变更执行完成 / When 测试工程师验证 / Then 验证结论记录在变更单中，所有"suspect"标记根据验证结果清除或保留
- Given 变更全流程完成 / When 项目经理关闭变更 / Then 变更单归档，审计日志完整记录全过程

#### 7.4.5 追溯链自动更新

**功能描述**：变更执行后，自动更新追溯关系。

**验收标准**：
- Given 变更删除了某条下游需求 / When 变更执行 / Then 上游需求的正向追溯自动移除该关联，追溯断裂检测重新执行
- Given 变更新增了某条下游需求 / When 变更执行 / Then 新追溯关系自动建立，法规关联自动继承
- Given 变更修改了需求层级 / When 变更执行 / Then 追溯矩阵自动更新，相关基线对比报告自动刷新

### 7.5 合规管理

> **设计原则**：合规是系统核心而非附加功能。合规驱动是系统成功的底层逻辑（INS-2）。

#### 7.5.1 电子签名（FR-0.2）

**功能描述**：符合21 CFR Part 11电子签名要求。

**签名要求**：

| 操作类型 | 签名含义 | 签名方式 |
|---------|---------|---------|
| 需求创建 | "我是本文档的创建者" | 用户名+密码确认+签名含义 |
| 需求审批 | "我批准此需求" | 用户名+密码确认+签名含义 |
| 评审通过 | "我评审并同意此需求" | 用户名+密码确认+签名含义 |
| 变更审批 | "我批准此变更" | 用户名+密码确认+签名含义 |
| 基线确认 | "我确认此基线" | 用户名+密码确认+签名含义 |
| 问题报告关闭 | "我确认问题已解决" | 用户名+密码确认+签名含义 |

**21 CFR Part 11电子签名合规矩阵（新增）**：

> **变更记录**：2026-05-22 | 许清楚 | 新增 | P-ORG-10：补充21 CFR Part 11电子签名合规矩阵

| Part 11 要求 | Med-RMS 方案 | 实现方式 |
|---|---|---|
| 11.50 签署显现 | ✅ | 签名记录包含签名者姓名、签名日期时间、签名含义，三者不可分离显示 |
| 11.70 签名与记录链接 | ✅ | 签名值=非对称加密(签名者ID+签名含义+签名时间+被签名文档哈希)，签名与记录不可裁切/复制/转移 |
| 11.100 签名唯一性 | ✅ | 每次签名操作需重新输入密码验证身份，签名记录全局唯一（UUID） |
| 11.200 双组件签名 | ✅ | 用户名（所知）+密码（所知），满足基本双组件要求 |
| 11.300 令牌/密码控制 | ✅ | 密码策略：长度≥8位+含大小写数字+90天更换+5次失败锁定30分钟+过期停用 |

**签名值计算规则（新增）**：
- 签名值 = RSA-Sign(签名者ID + 签名含义 + 签名时间戳 + 被签名实体类型 + 被签名实体ID + 被签名实体内容哈希SHA-256)
- 签名时弹出独立对话框，要求输入密码进行二次认证
- 签名时间使用服务器时间（可信时间源），不使用客户端时间

**验收标准**：
- Given 用户执行需要电子签名的操作 / When 系统弹出签名对话框 / Then 用户输入密码和签名含义，系统验证身份后记录签名
- Given 电子签名已记录 / When 用户尝试否认签名 / Then 系统拒绝，签名不可删除、不可修改、不可否认
- Given 审计人员查看签名记录 / When 查看签名详情 / Then 可看到签名者、签名时间、签名含义、签名时文档哈希值、签名值
- Given 签名后的文档被修改 / When 审计人员校验 / Then 签名值与文档哈希不匹配，系统标记为"签名失效"

#### 7.5.2 审计追踪（FR-0.2）

**功能描述**：符合21 CFR Part 11审计追踪要求。

**审计日志记录范围**：

**审计日志防篡改机制（修订）**：

> **变更记录**：2026-05-22 | 许清楚 | 修改 | A-RED-03：审计日志增加哈希链保护和数据库层强制约束

1. **哈希链机制**：每条审计日志记录包含 prev_hash（前一条记录的SHA-256哈希）和 current_hash，形成链式结构。任何篡改将导致后续所有记录的哈希校验失败
2. **数据库层强制保护**：通过数据库触发器阻止对审计日志表的UPDATE和DELETE操作
3. **定期完整性校验**：系统支持一键校验哈希链完整性，校验记录本身纳入审计日志
4. **保留期限**：审计日志至少保留15年（ISO 13485要求≥产品生命周期+1年，医疗器械产品生命周期可达10-15年）

**验收标准**：
- Given 任意系统操作发生 / When 操作完成 / Then 审计日志自动记录，延迟≤1秒
- Given 审计日志已记录 / When 任何用户尝试删除或修改日志 / Then 系统拒绝操作（数据库触发器阻止）
- Given 合规专员查询审计日志 / When 按时间范围/操作人/事件类型筛选 / Then 系统返回匹配日志，支持导出
