## 九、待修复架构缺陷

### ✅ 缺陷 1：事件驱动架构【R111 修订：已实现】

| 项目 | 详情 |
|------|------|
| **设计要求** | Transactional Outbox 表 + Debezium CDC，req-mgr 发布领域事件异步通知 trace-mgr / chg-mgr |
| **实际实现（R111 验证）** | **已实现 in-process Outbox 模式**：<br>• `med-rms-common/outbox/DomainEvent.java`（事件对象）<br>• `med-rms-common/outbox/OutboxMessage.java`（DB 实体）<br>• `med-rms-common/outbox/OutboxService.java`（含 `append()` 事务内落库 + `subscribe()` 注册订阅者 + `publishPending()` 30 秒定时器分发）<br>• `MedRmsApplication.java` 已加 `@EnableScheduling`<br>• 3 个模块已调用：`ChangeService`（3 处）/ `OaIntegrationService`（1 处）/ `TraceabilityService`（1 处） |
| **与设计差异** | 实际为 in-process 同步订阅者模式（无 Kafka / Debezium），适合当前单实例部署。设计上假设异步消息总线 |
| **影响** | 模块耦合度降低（异步解耦），但同进程事件总线限制了横向扩展 |
| **后续升级路径** | 部署 Kafka 后，可将 `OutboxService.subscribe()` 替换为 `KafkaTemplate.send()`，无需改动业务代码 |
