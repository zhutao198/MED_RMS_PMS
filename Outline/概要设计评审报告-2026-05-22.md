**评审基线**：PRD v2.1 + 系统架构 v1.1  
**评审对象**：Outline 目录下 7 份概要设计文档（v1.0）

## 评审总览

| 维度 | 评分 | 说明 |
|------|------|------|
| **需求可追溯性** | ★★★★☆ | PRD 功能点基本覆盖，少数功能缺失或降级 |
| **架构一致性** | ★★★★☆ | 核心架构决策与系统架构文档一致，少量偏差 |
| **数据模型完整性** | ★★★☆☆ | 与系统架构数据模型存在显著差异，需重点修订 |
| **接口设计完备性** | ★★★★☆ | API 覆盖率良好，少量接口缺失 |
| **合规合规性** | ★★★★☆ | 21 CFR Part 11 / IEC 62304 覆盖较全，细节待补充 |
| **文档间一致性** | ★★★☆☆ | 概要设计各文档间及与上游基线间存在不一致 |

**综合结论**：概要设计整体方向正确，架构决策与 PRD/系统架构对齐度高，但数据模型层与系统架构文档存在系统性差异（分层子表消失、部分表结构不一致），需在详细设计前完成修订。

## 一、严重问题（Critical）—— 必须修订后才能进入详细设计

### C-01：需求主表数据模型与系统架构 v1.1 严重不一致

**位置**：03-数据库概要设计 §2.1 `requirement` 表 vs 系统架构 §6.1

**问题描述**：

系统架构 v1.1 经过评审修订（A-RED-02），明确采用 **闭包表+分层子表（CTI）** 存储四层需求模型，包含以下核心表：
- `t_requirement`（公共主表）
- `t_user_requirement`（URS 子表）
- `t_product_requirement`（PRS 子表）
- `t_system_requirement`（SRS 子表）
- `t_design_requirement`（DRS 子表）
- `t_requirement_ancestor`（闭包表）
- `t_requirement_version`（版本历史表）
- `t_test_case`（测试用例表）
- `t_requirement_testcase`（需求-测试用例关联表）

但概要设计中：
1. **分层子表完全消失**：`requirement_version` 表被定义为"需求版本表（分层子表 CTI）"，包含 `level(URS/PRS/SRS/DRS)` + `content_json` 字段——这与架构文档的 CTI 方案完全矛盾。架构文档已明确 **消除 JSONB extended_fields**，各层级特有字段应在子表中以强类型列存储
2. **需求主表字段缺失**：架构定义的 `safety_class`、`requirement_category`、`baseline_id`、`is_deleted` 字段在概要设计中均未出现
3. **闭包表字段不一致**：架构定义 `t_requirement_ancestor(descendant_id, ancestor_id, depth)`，概要设计为 `requirement_closure(id, ancestor_id, descendant_id, depth, path_length)`，多出 `id` 和 `path_length` 字段，且命名不一致
4. **测试用例实体缺失**：架构定义的 `t_test_case` 和 `t_requirement_testcase` 表在概要设计中完全缺失

**影响**：数据模型是详细设计和编码的基石，此不一致将直接导致开发人员按错误模型实现。

**修订建议**：
1. 将 `requirement_version` 拆分为 `requirement`（公共主表）+ 4 张分层子表，严格对齐系统架构 §6.1
2. 补充缺失字段 `safety_class`、`requirement_category`、`baseline_id`、`is_deleted`
3. 统一闭包表命名为 `requirement_ancestor`，移除多余的 `id` 和 `path_length`
4. 补充 `test_case` 和 `requirement_testcase` 两张表

### C-02：需求编号生成规则与 PRD 定义不一致

**位置**：03-数据库概要设计 §6.2 触发器 vs PRD §7.1.2~7.1.5

**问题描述**：

PRD 明确定义了四层需求的编号格式：
- URS：`URS-{项目编号}-{序号}`
- PRS：`PRS-{项目编号}-{序号}`
- SRS：`SRS-{项目编号}-{序号}`
- DRS：`DRS-{项目编号}-{序号}`

概要设计的触发器生成的编号格式为 `REQ-{YYYY}-{NNNN}`，与 PRD 定义完全不同。

**影响**：编号是需求的核心标识，格式不一致将导致用户困惑和追溯链路展示异常。

**修订建议**：按需求层级生成不同前缀的编号（URS/PRS/SRS/DRS），使用项目编号+序号而非年份+序号。

### C-03：需求状态机与 PRD 定义存在关键差异

**位置**：02-核心业务流程设计 §1.1~1.2 vs PRD §7.1.2~7.1.5

**问题描述**：

| 差异点 | PRD 定义 | 概要设计定义 | 影响 |
|--------|---------|-------------|------|
| PRS 额外状态 | 待拆解、已拆解 | 未定义 | 拆解工作台核心流程缺失 |
| SRS 额外状态 | 待拆解、已拆解、待验证、已验证 | 未定义 | 测试验证流程不完整 |
| DRS 完整状态链 | 实现中→已实现→待验证→已验证→已关闭 | 仅定义到 Verified→Closed | DRS 的实现-验证闭环缺失 |
| 基线化状态 | 已基线化（URS/PRS/SRS/DRS 通用） | 未定义 | 基线管理流程不完整 |
| 已变更状态 | 已变更→重新评审 | 仅定义 ChangeRequested | 变更后的处理流程不一致 |

**影响**：状态机是业务流程的核心骨架，差异将导致前端页面、后端 API、审计日志均按错误逻辑实现。

**修订建议**：按 PRD §7.1.2~7.1.5 逐层细化状态机定义，补充缺失状态和转换。

---

### C-04：审计日志保留策略与系统架构定义矛盾

**位置**：03-数据库概要设计 §8 vs 系统架构 §6.3.1

**问题描述**：

系统架构 v1.1 明确规定：**审计日志至少保留 15 年**（ISO 13485 要求+医疗器械产品生命周期）。

概要设计 §8 数据归档策略中：
- 审计日志（audit_log）：标注"永久在线"、"永不删除"——与架构一致
- 签名记录（signature_record）：标注"永久在线"、"永不删除"——与架构一致

但系统架构中审计日志表有 `reason` 字段（变更/签名时必填），概要设计中 audit_log 核心字段未包含 `reason`。21 CFR Part 11 11.10(e) 明确要求审计日志必须记录操作原因。

**修订建议**：audit_log 表补充 `reason` 字段，并标注为"变更/签名操作时必填"。

## 二、重要问题（Major）—— 建议在详细设计前修订

### M-01：变更管理优先级与 PRD/架构不一致

**位置**：01-业务模块划分 §8 vs PRD §6 需求池

**问题描述**：

PRD 需求池中，变更管理相关需求分布在 P0 和 P1：
- P0：FR-0.10 变更影响自动标记（suspect）
- P1：FR-1.3 需求变更自动影响评估、FR-1.7 变更审批流在线化

概要设计 §8 将 chg-mgr 标注为 **P1** 优先级，但 FR-0.10 是 P0 需求。概要设计的变更请求表和影响分析表仍放在 chg_schema（P1 模块），而 suspect 标记功能需要 P0 阶段可用。

**修订建议**：将 suspect 标记相关能力（下游自动标记）拆分到 req-mgr 的 P0 阶段实现，或在 chg-mgr 内标注 P0 子功能的交付阶段。

---

### M-02：电子签名数据模型与系统架构存在差异

**位置**：03-数据库概要设计 §2.5 vs 系统架构 §6.4

**问题描述**：

| 字段 | 系统架构定义 | 概要设计定义 | 差异 |
|------|-------------|-------------|------|
| 签名唯一标识 | `signature_id UUID UNIQUE NOT NULL` | 无此字段 | 缺少全局唯一标识 |
| 签名含义编码 | `meaning_code` + `meaning_display` | 仅有 `intent` | PRD Part 11 合规要求签名含义不可扩展 |
| 被签名实体哈希 | `entity_hash CHAR(64) NOT NULL` | `document_hash` | 命名不一致，架构更精确 |
| 认证方式 | `auth_method PASSWORD/OTP` | 无此字段 | 缺少认证方式记录 |
| IP 地址 | `ip_address` | 无此字段 | 21 CFR Part 11 审计要求 |
| 签名有效性 | `is_valid BOOLEAN` | `status(Pending/Completed/Invalidated)` | 状态机 vs 布尔值，架构方案更清晰 |
| 签名验证记录 | 独立 `t_signature_verification` 表 | 无 | 签名验证结果无法追溯 |

概要设计额外引入了 `signature_intent`（签名意图表）和 `otp_challenge`（OTP 挑战表），这些在架构文档中未定义，但属于概要设计的合理细化，可保留。

**修订建议**：对齐系统架构 §6.4 的字段定义，补充缺失字段，统一命名。

---

### M-03：SOUP 管理数据模型与 PRD/架构定义不一致

**位置**：03-数据库概要设计 §2.4 `soup_record` vs 系统架构 §6.5 `t_soup_component`

**问题描述**：

系统架构 `t_soup_component` 定义的核心字段：
- `component_name`、`supplier`、`version`、`license_type`、`known_anomalies`、`safety_impact`、`project_id`、`last_reviewed_at`

PRD §7.1.10 额外定义了：
- `SOUP-ID`、`用途描述`、`关联需求`、`异常审查记录`、`状态（登记/已评估/已审查/已批准）`

概要设计 `soup_record` 的字段：
- `name, version, supplier, purpose, license, safety_class, last_review_date, review_result`

**差异**：
1. PRD 的"关联需求"（多选关联）在概要设计中缺失
2. PRD 的"异常审查记录"字段在概要设计中缺失
3. PRD 的状态定义（登记/已评估/已审查/已批准）与概要设计不一致
4. 命名不一致：架构 `known_anomalies` vs 概要设计未明确

**修订建议**：补充关联需求、异常审查记录、完整状态定义，统一命名。

---

### M-04：缺少问题报告管理的数据模型和接口

**位置**：03-数据库概要设计、04-接口概要设计

**问题描述**：

PRD §7.1.11 明确定义了问题报告管理（FR-0.14），包括：
- 问题报告字段：PR-ID、问题描述、严重度、发现日期、根本原因分析、关联变更请求、不行动理由、验证确认、状态
- 验收标准：Critical/Major 自动通知、不行动理由必填等

系统架构中问题报告也作为 IEC 62304 Clause 9 的重要功能。

但概要设计中：
1. **数据库**：无问题报告相关表定义
2. **接口**：无问题报告相关 API 定义
3. **流程**：02-核心业务流程设计中无问题报告流程
4. **菜单**：05-页面与菜单结构设计中无问题报告页面

**修订建议**：在 compliance_schema 或独立 schema 中补充 `problem_report` 表，补充 CRUD API、流程定义和页面路由。

---

### M-05：基线管理功能在概要设计中不完整

**位置**：02-核心业务流程设计、04-接口概要设计

**问题描述**：

PRD FR-0.8 和 US-15 明确定义了基线管理需求，系统架构 §6.6 定义了基线数据模型。概要设计中：
1. 数据库有 `baseline` 和 `baseline_item` 表定义（✅）
2. API 有基线创建/锁定/解锁/对比接口（✅）
3. **但缺少**：基线锁定需"QA经理+项目经理双人电子签名"的流程定义（PRD 要求）
4. **但缺少**：基线对比的具体算法说明（差异计算逻辑）
5. 05-页面设计中的基线签名流程仅简单提及"签名锁定"

**修订建议**：在 02-核心业务流程设计中补充基线锁定/解锁的完整流程（含双人签名），在接口设计中明确基线对比的返回结构。

### M-06：DCP 门控条件与 PRD 定义不一致

**位置**：02-核心业务流程设计 §9.2 vs PRD §7.1.7

**问题描述**：

| DCP | PRD 校验规则 | 概要设计校验规则 | 差异 |
|-----|-------------|-----------------|------|
| DCP1 | 所有URS必须"已通过"；每条URS至少关联1条法规条款 | 项目信息完整、团队成员已分配 | 概要设计缺少 URS 通过率和法规关联校验 |
| DCP2 | 所有URS已拆解为PRS；所有PRS必须"已通过"；追溯完整率≥90% | URS全部Created、初始风险已识别 | "Created"≠"已通过"，追溯率未校验 |
| DCP3 | 所有PRS已拆解为SRS；所有SRS已拆解为DRS；DRS实现完成率≥80% | URS→PRS/SRS追溯率≥80%、PRS/SRS已Approved | 追溯率阈值不一致（90% vs 80%），缺少 DRS 实现完成率校验 |
| DCP4 | DRS验证通过率≥95%；测试覆盖率100%；追溯闭环率≥95% | 追溯覆盖率≥95%、验证记录已签名 | 缺少 DRS 验证通过率和测试覆盖率校验 |
| DCP5 | 全链路追溯闭环率≥99%；合规证据包可生成；无高风险未闭环项 | 全部需求Closed/Retired、无高风险未关闭、审计日志完整 | 追溯闭环率缺失（99%），合规证据包未校验 |

**修订建议**：严格按 PRD §7.1.7 的 DCP 门控条件清单修订，补全所有校验项和阈值。

---

### M-07：Outbox 表定义缺失

**位置**：03-数据库概要设计

**问题描述**：

系统架构 §3.5.2 明确定义了 Transactional Outbox 表结构（`event_id, aggregate_type, aggregate_id, event_type, payload, created_at, published`），且这是跨模块通信的核心机制。

概要设计中：
1. 数据库部分 **完全没有 Outbox 表的定义**
2. 04-接口概要设计 §5.2 事件清单中列出了领域事件，但未说明事件如何持久化和投递
3. 01-业务模块划分 §5.1 描述了 Outbox + CDC 流程，但未关联到具体的表定义

**修订建议**：在公共 Schema 或各模块 Schema 中补充 Outbox 表定义，与系统架构 §3.5.2 对齐。

---

### M-08：JWT 黑名单持久化表缺失

**位置**：03-数据库概要设计

**问题描述**：

系统架构 §5 ADR-007 明确要求：**JWT 黑名单同时持久化到数据库表 `t_jwt_blacklist`（非仅 Redis），确保审计可追溯**。

概要设计中 sys_schema 包含 12 张表，但无 `jwt_blacklist` 表。如果 JWT 黑名单仅存 Redis，Redis 重启后黑名单丢失，无法满足审计追溯要求。

**修订建议**：在 sys_schema 中补充 `jwt_blacklist` 表。

## 三、一般问题（Minor）—— 建议在开发前修订

### m-01：前端技术版本与系统架构不一致

**位置**：05-页面与菜单结构设计 §7.1 vs 系统架构 §4.1

|------|-------------|-------------|
| Vue 3 | 3.5.x | 3.4.x |
| Element Plus | 2.9.x | 2.7.x |
| Vite | 6.x | 5.x |

**修订建议**：统一为系统架构定义的版本。

### m-02：需求优先级枚举值不一致

**位置**：03-数据库概要设计 §4.1 字典 `REQ_PRIORITY` vs PRD §7.1.2

**差异**：
- PRD 定义：必须/应该/可选（MoSCoW 原则）
- 概要设计字典：P0-必须, P1-重要, P2-一般, P3-低

PRD 使用 MoSCoW（MUST/SHOULD/COULD），概要设计使用 P0~P3 数字分级，两者语义不同。

**修订建议**：统一为 MoSCoW 枚举（MUST/SHOULD/COULD），与 PRD 和系统架构对齐。

---

### m-03：签名值计算方式不一致

**位置**：02-核心业务流程设计 §5.2 vs 系统架构 §5 ADR-008 vs PRD §7.5.1

**差异**：
- 系统架构：`SHA-256(signer_id + meaning_code + signed_at_iso8601 + entity_hash)`
- 概要设计：`SHA-256(document_hash + user_id + timestamp + intent)`
- PRD：`RSA-Sign(签名者ID + 签名含义 + 签名时间戳 + 被签名实体类型 + 被签名实体ID + 被签名实体内容哈希SHA-256)`

三处定义各不相同：系统架构用 SHA-256，PRD 用 RSA-Sign，概要设计用 SHA-256 但字段名不同。

**修订建议**：在概要设计阶段统一签名值计算方式，建议使用系统架构的 SHA-256 方案（满足合规且实现简单），并在 PRD 中同步更新。

---

### m-04：审计日志 `action` 字段与系统架构 `event_type` 不一致

**位置**：03-数据库概要设计 §2.4 `audit_log` 核心字段

**差异**：
- 系统架构：`event_type`（CREATE/MODIFY/DELETE/STATUS_CHANGE/SIGN/LOGIN）
- 概要设计：`action`（无枚举定义）

**修订建议**：统一字段名为 `event_type`，明确枚举值域。

### m-05：缺少 IEC 62304 合规检查清单的实现

**位置**：PRD FR-0.15、US-9

**问题描述**：

PRD FR-0.15 和 US-9 明确定义了 IEC 62304 合规检查清单功能（Clause 5-9 覆盖状态显示），但概要设计中：
1. 无合规检查清单相关数据表
2. 无合规检查清单 API
3. 无合规检查清单页面路由

**修订建议**：在 compliance_schema 中补充 `iec62304_checklist` 表（或复用 `regulatory_mapping` 表），补充 API 和页面。

### m-06：RPO/RTO 与系统架构不一致

**位置**：06-非功能与规则约束 §8.1 vs 系统架构 §5 ADR-012

**差异**：
- 系统架构：RTO ≤ 30分钟，RPO ≤ 5分钟
- 概要设计：RPO ≤ 1小时，RTO ≤ 4小时

概要设计的容灾指标远低于系统架构要求。

**修订建议**：对齐系统架构 ADR-012 的 RTO/RPO 指标。

### m-07：部署架构描述不一致

**位置**：06-非功能与规则约束 §12.1 vs 系统架构 §9.1

**差异**：
- 系统架构：Docker Compose 单机主备方案，PostgreSQL 主备流复制
- 概要设计：Docker + Kubernetes 2副本部署，Redis Sentinel 1主2从3哨兵，MinIO 分布式4节点

概要设计的部署规格远超系统架构的定位（<200用户、小团队），且 K8s 增加运维复杂度，与"模块化单体优先"的简化原则矛盾。

**修订建议**：初期采用 Docker Compose 单机方案（对齐系统架构），预留 K8s 演进路径。

---

### m-08：需求来源枚举不一致

**位置**：03-数据库概要设计 §4.1 字典 `REQ_SOURCE` vs PRD §7.1.2

**差异**：
- PRD：客户/市场/法规/内部/竞品
- 概要设计：用户/法规/标准/风险/市场/内部

PRD 有"客户"和"竞品"，概要设计有"标准"和"风险"，双方各有缺失。

**修订建议**：合并为完整枚举：用户/客户/市场/法规/标准/风险/内部/竞品。

---

### m-09：操作序列强制检查规则未体现

**位置**：PRD FR-0.17、系统架构 §8.3

**问题描述**：

PRD FR-0.17 和系统架构 §8.3 明确定义了操作序列强制检查（如审批前必须有评审记录），对应 21 CFR Part 11 11.10(f)。

概要设计中仅在 06-非功能与规则约束 §3.3 通用校验清单的"提交评审"和"变更审批"行中简单提及，缺乏系统性的操作序列校验规则定义。

**修订建议**：在 06-非功能与规则约束中补充完整的操作序列强制检查矩阵，覆盖所有关键业务操作的前置条件。

### m-10：缺少 NMPA eRPS 报告导出功能

**位置**：PRD FR-1.12、US-14

**问题描述**：

PRD FR-1.12 和 US-14 定义了 NMPA eRPS 格式报告导出功能（中英文双语），概要设计中仅在 04-接口概要设计 §2.8 有通用的 `POST /api/v1/reports/export`，未明确 NMPA eRPS 格式支持。

**修订建议**：在报表导出 API 中增加 `format` 参数，支持 `NMPA_ERPS` 格式，补充模板定义。

## 四、建议项（Suggestion）—— 可在后续迭代中优化

### S-01：概要设计文档间交叉引用不足

7 份概要设计文档相对独立，缺少文档间的交叉引用。例如：
- 03-数据库概要设计中的表应在 04-接口概要设计中引用
- 02-核心业务流程设计中的流程应在 05-页面设计中映射
- 07-架构决策文档中的引用关系可更完整

**建议**：增加文档间的引用标注，便于开发人员快速定位关联内容。

### S-02：前端缺少拆解工作台页面设计

PRD §7.2.5 定义了"拆解工作台"为核心工作界面，系统架构 §3.4 也定义了 `DecompositionWorkbench` 组件。但 05-页面设计中无独立的拆解工作台路由和布局设计，仅在需求详情页中有简单的子需求列表。

**建议**：补充拆解工作台的页面布局设计（树形视图+拆解面板+覆盖率校验）。

### S-03：缺少通知机制的设计

PRD §7.9 定义了通知机制（多角色订阅、站内信+邮件），概要设计中仅在 Redis 缓存中定义了 `esign:otp` 键，未设计通知系统的数据模型、API 和页面。

**建议**：在 sys_schema 中补充通知相关表，补充通知 API。

### S-04：错误码体系可补充合规相关错误

04-接口概要设计中的错误码体系覆盖了基本业务场景，但缺少：
- 哈希链校验失败的详细错误码（仅有 CP0301 一个）
- OTP 发送频率限制错误码
- 签名密码策略错误码（复杂度不足、历史重复等）
- DCP 门控校验各条件的细粒度错误码

**建议**：补充合规场景的细粒度错误码。

### S-05：数据库表缺少 `updated_by` / `updated_at` 审计字段

系统架构 §6.1.1 `t_requirement` 定义了 `updated_by` 和 `updated_at` 字段，概要设计中部分表（如 `requirement`、`change_request`）缺少这两个审计字段。

**建议**：为所有可修改的业务表补充 `updated_by` 和 `updated_at` 字段。

## 七、评审结论

### 总体评价

概要设计文档体系完整、结构清晰，7 份文档覆盖了业务模块、流程、数据库、接口、页面、非功能和架构决策等核心维度。与 PRD v2.1 和系统架构 v1.1 的整体方向一致，9 个限界上下文划分、模块化单体架构、事务性发件箱、哈希链审计日志等核心架构决策均得到了正确贯彻。

### 核心风险

**数据模型层是最大风险点**——概要设计与系统架构在需求主表模型上存在系统性差异（分层子表消失、字段缺失、命名不一致），如果不在详细设计前修订，将导致开发阶段的大面积返工。

### 修订建议

1. **优先处理 4 个严重问题（C-01~C-04）**——这是进入详细设计的前置条件
2. **其次处理 8 个重要问题（M-01~M-08）**——影响功能完整性和合规性
3. **开发前完成 10 个一般问题（m-01~m-10）**——确保一致性
4. **建议项（S-01~S-05）可在迭代中逐步完善**
