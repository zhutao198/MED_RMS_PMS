## 1. 架构决策记录（ADR Summary）

> 以下汇总了系统架构文档中定义的 12 个 ADR，并补充了概要设计层面的决策依据和影响分析。

### ADR-001：采用模块化单体架构（Modular Monolith）

| 维度 | 内容 |
|------|------|
| **决策** | 采用 Spring Modulith 实现模块化单体，而非直接微服务 |
| **背景** | 团队规模中等（5-8人），业务域边界清晰但初期流量不大 |
| **理由** | ① 避免微服务带来的分布式复杂度 ② 模块间通过事件通信，未来可拆 ③ 单库多Schema简化运维 ④ 开发阶段迭代速度快 |
| **影响** | 每个模块必须严格遵守独立Schema，禁止跨模块直接表关联 |
| **可逆性** | 高——模块可逐步拆分为独立微服务 |

### ADR-002：DDD 限界上下文划分

| 维度 | 内容 |
|------|------|
| **决策** | 划分为 9 个限界上下文：需求、追溯、变更、合规、电子签名、风险、项目、报表、系统 |
| **背景** | 医疗器械需求管理涉及需求、合规、变更等多个业务域 |
| **理由** | ① 每个上下文有独立的通用语言 ② 遵循 IEC 62304 条款映射 ③ 变更频率不同（系统管理稳定，需求管理高频） ④ 权限边界自然对齐 |
| **影响** | 电子签名作为独立上下文，与JWT认证完全解耦 |

### ADR-003：闭包表+分层子表（CTI）存储需求层级

| 维度 | 内容 |
|------|------|
| **决策** | 使用闭包表（Closure Table）+ 分层子表（Class Table Inheritance）存储四层需求模型 |
| **备选方案** | ① 单表继承（STI）——查询简单但字段冗余 ② 路径枚举（Path Enumeration）——层级深时更新复杂 |
| **理由** | ① 闭包表支持任意深度层级查询，O(1)查祖先/子孙 ② CTI为每层需求独立子表，避免空字段 ③ 追溯矩阵需要快速遍历层级 |
| **影响** | 需求创建时需同时写入闭包表（ancestor=self, depth=0） |

### ADR-004：事务性发件箱模式（Transactional Outbox + Debezium CDC）

| 维度 | 内容 |
|------|------|
| **决策** | 跨模块事件使用 Outbox + CDC 投递，替代 Spring ApplicationEvent |
| **备选方案** | ① Spring ApplicationEvent——同事务，失败影响业务 ② 消息队列（Kafka/RabbitMQ）——增加运维复杂度 |
| **理由** | ① 事件写入与业务操作同一事务，保证不丢 ② Debezium CDC自动捕获变更，无需额外写入 ③ 初期不需要Kafka的全量能力 |
| **影响** | 每个模块需要一张 outbox 表；Debezium 需要独立部署 |

### ADR-005：审计日志哈希链

| 维度 | 内容 |
|------|------|
| **决策** | 审计日志采用 prev_hash/current_hash 哈希链 + 数据库触发器阻止 UPDATE/DELETE |
| **理由** | 21 CFR Part 11 要求审计日志不可篡改 ① 哈希链使任何修改均可检测 ② 触发器作为硬性保障 ③ 定时校验作为发现机制 |
| **影响** | 审计日志表不可UPDATE/DELETE，只可INSERT |

### ADR-006：JWT 与电子签名分离

| 维度 | 内容 |
|------|------|
| **决策** | JWT 仅用于 API 访问认证，电子签名使用独立的二次认证（签名密码+OTP） |
| **理由** | ① 21 CFR Part 11 要求签名者需明确表达签名意图 ② JWT 不会每次都验证签名者身份 ③ 签名密码+OTP 提供"明确的签名行为"证据 |
| **影响** | 签名场景需要额外的密码输入和OTP验证步骤 |

### ADR-007：CQRS Lite 用于仪表盘

| 维度 | 内容 |
|------|------|
| **决策** | 仪表盘查询采用 CQRS Lite 模式——异步更新统计快照表，查询直接读快照 |
| **备选方案** | ① 实时聚合查询——大数据量性能差 ② 完整CQRS——读写模型完全分离，过度设计 |
| **理由** | ① 仪表盘数据允许15分钟延迟 ② 统计快照表减少复杂JOIN ③ 实现简单，维护成本低 |
| **影响** | statistics_snapshot 表需要定期更新（事件驱动+定时兜底） |

### ADR-008：RBAC 权限模型

| 维度 | 内容 |
|------|------|
| **决策** | 采用 RBAC（基于角色的访问控制），8类内置角色 |
| **备选方案** | ① ABAC（基于属性的访问控制）——灵活但复杂 ② ACL（访问控制列表）——对象级控制，管理繁琐 |
| **理由** | ① 医疗器械领域角色定义明确 ② RBAC简单直观，满足权限需求 ③ 8类角色覆盖全部使用场景 |
| **影响** | 如需更细粒度的数据权限，需在Service层额外编码实现 |

### ADR-009：泛微 OA 集成策略

| 维度 | 内容 |
|------|------|
| **决策** | OAuth2 授权码模式 SSO + 双向 API 审批流 + 定时组织架构同步 |
| **理由** | ① 企业已有泛微OA，用户习惯OA登录 ② 重大变更审批需要OA审批流 ③ 组织架构以OA为准，单向同步 |
| **影响** | 需要维护OA API对接的适配层；OA不可用时SSO降级为本地登录 |

### ADR-010：PostgreSQL 多Schema 隔离

| 维度 | 内容 |
|------|------|
| **决策** | 同一数据库实例使用9个Schema隔离各模块数据 |
| **备选方案** | ① 每模块独立数据库——运维复杂 ② 单Schema+前缀——隔离性差 |
| **理由** | ① 单实例管理简单 ② Schema级别权限隔离 ③ 跨Schema查询显式指定，避免耦合 ④ 演进为微服务时可逐Schema拆出 |
| **影响** | 跨模块关联只能用弱引用（存ID），不建外键 |

### ADR-011：前端 Vue 3 + Element Plus

| 维度 | 内容 |
|------|------|
| **决策** | 前端采用 Vue 3 (Composition API) + Element Plus + Vite |
| **理由** | ① Vue 3 Composition API 代码复用性好 ② Element Plus 企业级组件丰富 ③ Vite 开发体验快 ④ 团队有Vue技术栈经验 |
| **影响** | 部分定制组件需自行开发（追溯矩阵、风险矩阵等） |

### ADR-012：Spring Boot 3.3.x + Java 17

| 维度 | 内容 |
|------|------|
| **决策** | 后端采用 Spring Boot 3.3.x + Java 17 + MyBatis-Plus |
| **理由** | ① Spring Boot 3.x 支持虚拟线程和 GraalVM ② Java 17 LTS 长期支持 ③ MyBatis-Plus 对复杂查询友好 ④ Spring Modulith 依赖 Spring Boot 3.x |
| **影响** | 需要团队Java 17升级培训 |

## 2. 概要设计层面的补充决策

### SD-001：乐观锁并发控制

| 维度 | 内容 |
|------|------|
| **决策** | 所有核心业务实体（Requirement, ChangeRequest, RiskItem等）使用乐观锁（version字段） |
| **理由** | 医疗器械领域并发冲突概率低，乐观锁性能优于悲观锁 |
| **规则** | 更新时 WHERE id = ? AND version = ?，version不匹配返回错误码 RQ0600 |

### SD-002：软删除策略

| 维度 | 内容 |
|------|------|
| **决策** | 仅系统管理模块使用软删除（deleted字段），业务数据不使用软删除 |
| **理由** | ① 合规要求所有业务操作有审计日志，删除操作记录在日志中 ② 需求/变更等核心数据不可删除，只能归档/退役 |
| **例外** | Draft 状态的需求可物理删除（尚未进入正式流程） |

### SD-003：文件存储策略

| 维度 | 内容 |
|------|------|
| **决策** | 文件附件使用 MinIO 对象存储，数据库仅存储文件ID和元信息 |
| **规则** | ① 文件名原始保留，MinIO内使用UUID重命名 ② 文件大小限制：单文件≤100MB，单需求≤500MB ③ 合规文档不允许删除，仅可追加新版本 |

### SD-004：定时任务框架

| 维度 | 内容 |
|------|------|
| **决策** | 使用 Spring Scheduler + 数据库分布式锁 |
| **任务清单** | ① 哈希链校验（每日02:00） ② OA组织架构同步（每日06:00） ③ 统计快照更新（每15分钟） ④ OTP清理（每日03:00） ⑤ 过期评审提醒（每日09:00） ⑥ 密码过期提醒（每日08:00） |

### SD-005：异常处理策略

| 维度 | 内容 |
|------|------|
| **决策** | 分层异常处理 + 全局异常转换 |
| **规则** | ① 业务异常 → 自定义BusinessException，携带错误码 ② 参数校验异常 → 统一转换为SY01xx ③ 权限异常 → 统一转换为SY04xx ④ 未知异常 → SY0800，记录ERROR日志 ⑤ 全局 @RestControllerAdvice 统一响应格式 |

## 3. 术语表与缩写表

### 3.1 业务术语

| 术语 | 全称 | 定义 |
|------|------|------|
| URS | User Requirement Specification | 用户需求规格，描述用户/业务角度的需求 |
| PRS | Product Requirement Specification | 产品需求规格，描述产品应具备的功能 |
| SRS | Software Requirement Specification | 软件需求规格，描述软件系统的功能和非功能需求 |
| DRS | Design Requirement Specification | 设计需求规格，描述软件设计的具体技术需求 |
| DCP | Decision Check Point | 决策检查点，IPD流程中的阶段门控 |
| IPD | Integrated Product Development | 集成产品开发，一种产品开发流程框架 |
| RPN | Risk Priority Number | 风险优先级数 = 严重度(S) × 概率(P) × 可检测性(D) |
| SOUP | Software of Unknown Provenance | 未知来源软件，包括第三方库和开源组件 |
| CTI | Class Table Inheritance | 类表继承，一种数据库继承映射模式 |
| RBAC | Role-Based Access Control | 基于角色的访问控制 |
| CQRS | Command Query Responsibility Segregation | 命令查询职责分离 |
| CDC | Change Data Capture | 变更数据捕获 |
| ACL | Anti-Corruption Layer | 防腐层，隔离外部模型 |
| OTP | One-Time Password | 一次性密码/动态验证码 |
| SSO | Single Sign-On | 单点登录 |
| WAL | Write-Ahead Log | 预写式日志（PostgreSQL） |
| BRIN | Block Range Index | 块范围索引（PostgreSQL） |
| DLQ | Dead Letter Queue | 死信队列 |

### 3.2 法规标准缩写

| 缩写 | 全称 | 说明 |
|------|------|------|
| IEC 62304 | IEC 62304:2006+A1:2015 | 医疗器械软件——软件生命周期过程 |
| ISO 13485 | ISO 13485:2016 | 医疗器械——质量管理体系 |
| 21 CFR Part 11 | Title 21 CFR Part 11 | 电子记录；电子签名 |
| ISO 14971 | ISO 14971:2019 | 医疗器械——风险管理 |
| NMPA | National Medical Products Administration | 国家药品监督管理局 |
| eRPS | Electronic Regulated Product Submission | 电子监管产品提交 |
