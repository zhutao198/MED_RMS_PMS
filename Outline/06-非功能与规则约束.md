## 1. 权限控制

### 1.1 RBAC 权限模型

```
User ──M:N──→ Role ──M:N──→ Permission ──1:N──→ Resource
```

**模型说明：**
- **User**：系统用户，通过泛微OA同步或手动创建
- **Role**：角色，分为系统内置角色和自定义角色
- **Permission**：权限点，细分为菜单权限、按钮权限、API权限
- **Resource**：资源，包括菜单、页面、API端点、数据对象

### 1.2 内置角色定义

| 角色编码 | 角色名称 | 职责范围 | 数据范围 |
|----------|----------|----------|----------|
| ADMIN | 系统管理员 | 全系统管理，用户/角色/配置/字典 | 全部项目 |
| PM | 项目经理 | 项目级管理，需求审批/基线/门控 | 所属项目 |
| RE | 需求工程师 | 需求创建/编辑/追溯 | 所属项目（自己创建的） |
| QA | QA工程师 | 评审/合规/审计/签名 | 所属项目 |
| DEV | 研发工程师 | 需求查看/实现标记/变更发起 | 所属项目 |
| REVIEWER | 评审人 | 需求评审/签名确认 | 被分配的需求 |
| SIGNER | 签名人 | 电子签名/审批确认 | 被分配的签名任务 |
| VIEWER | 只读用户 | 查看/导出 | 授权项目 |

### 1.3 数据权限规则

| 规则 | 说明 |
|------|------|
| 项目隔离 | 用户只能访问自己所属项目的数据 |
| 创建人权限 | 需求工程师可编辑自己创建的需求（Draft状态） |
| 角色继承 | 项目经理拥有需求工程师所有权限 |
| 全局角色 | 系统管理员跨项目访问，不受项目隔离限制 |
| 只读覆盖 | 只读用户对任何数据都只有查看和导出权限 |

### 1.4 权限校验层级

| 层级 | 校验位置 | 说明 |
|------|----------|------|
| 菜单级 | 前端路由守卫 | 无权限菜单不显示，路由不可访问 |
| 按钮级 | 前端指令 `v-permission` | 无权限按钮不渲染 |
| API级 | 后端 Spring Security | 无权限请求返回403 |
| 数据级 | 后端 MyBatis 拦截器 | 自动追加项目ID过滤条件 |

## 2. 登录与会话管理

### 2.1 认证体系

| 场景 | 认证方式 | Token类型 | 有效期 |
|------|----------|-----------|--------|
| 账号密码登录 | 用户名+密码 | JWT Access Token | 15分钟 |
| OA SSO登录 | OAuth2授权码 | JWT Access Token | 15分钟 |
| Token刷新 | Refresh Token | 新的Access Token | — |
| 电子签名 | 签名密码+OTP | — | 一次性 |

### 2.2 Token 管理

| 规则 | 说明 |
|------|------|
| Access Token | JWT格式，包含userId、roles、permissions，有效期15分钟 |
| Refresh Token | 不透明Token，存储于Redis，有效期24小时 |
| Token存储 | 前端存储于内存（非localStorage），防止XSS窃取 |
| Token刷新 | Access Token过期前5分钟自动刷新，使用Refresh Token |
| Token撤销 | 登出时将Refresh Token加入Redis黑名单 |
| 并发限制 | 同一用户最多5个活跃会话，超出时最早会话失效 |

### 2.3 密码策略

| 策略 | 要求 |
|------|------|
| 密码长度 | ≥8位 |
| 复杂度 | 大写+小写+数字+特殊字符，至少包含3种 |
| 历史限制 | 新密码不能与最近5次密码相同 |
| 锁定策略 | 连续5次错误锁定30分钟 |
| 过期策略 | 密码90天过期，过期前7天提醒 |
| 签名密码 | 独立于登录密码，≥6位，至少包含数字+字母 |

## 3. 数据校验

### 3.1 前端校验规则

| 字段类型 | 校验规则 | 说明 |
|----------|----------|------|
| 必填字段 | 非空校验 | 红色星号标记，实时校验 |
| 需求标题 | 1-200字符 | 不允许纯空格 |
| 需求描述 | 10-5000字符 | 支持富文本 |
| 需求编号 | 正则：`(URS\|PRS\|SRS\|DRS)-[A-Z0-9]+-\d{4}` | 系统自动生成，层级前缀+项目编号+序号，不可手动修改 |
| 优先级 | 枚举值校验 | MUST/SHOULD/COULD/WONT（MoSCoW模型） |
| 邮箱 | 邮箱格式正则 | 标准RFC格式 |
| 手机号 | 11位数字 | 中国大陆手机号 |
| 日期 | ISO 8601格式 | 开始日期≤结束日期 |
| 数字范围 | RPN: 1-125, S/P/D: 1-5 | 前后端双重校验 |

### 3.2 后端校验规则

| 校验层 | 框架 | 说明 |
|--------|------|------|
| 参数校验 | Jakarta Validation | @NotNull, @Size, @Pattern等注解 |
| 业务校验 | 业务Service层 | 状态机校验、权限校验、关联校验 |
| 数据完整性 | 数据库约束 | NOT NULL, UNIQUE, CHECK, 外键（同Schema内） |
| 并发控制 | 乐观锁 | version字段，更新时版本号+1 |

### 3.3 通用校验清单

| 场景 | 校验内容 |
|------|----------|
| 创建需求 | 标题唯一（同项目内）、类型+层级匹配、项目状态Active |
| 提交评审 | 需求状态为Draft/Rejected、必填字段完整、DCP校验通过 |
| 评审操作 | 评审人已分配、评审轮次未超限 |
| 创建追溯 | 源≠目标、链接不重复、目标需求状态允许 |
| 创建变更 | 变更原因非空、至少关联1个受影响需求 |
| 变更审批 | 影响分析已完成、审批人有权限 |
| 基线锁定 | 需求全部Approved、签名完成、无未关闭变更 |
| 电子签名 | 签名密码正确、OTP正确且未过期、签名意图有效 |
| DCP评审 | 门控条件全部满足（自动校验） |

## 4. 日志记录

### 4.1 日志分类

| 日志类型 | 存储位置 | 保留策略 | 说明 |
|----------|----------|----------|------|
| 审计日志 | compliance_schema.audit_log | 永久 | 21 CFR Part 11 合规，哈希链保护 |
| 操作日志 | sys_schema.operation_log | 1年在线 | 用户操作记录，非合规类 |
| 登录日志 | sys_schema.login_log | 6个月在线 | 登录/登出记录 |
| 应用日志 | 文件（ELK） | 30天滚动 | Spring Boot应用日志 |
| 访问日志 | 文件（ELK） | 7天滚动 | HTTP请求日志 |
| 审计日志（签名） | esign_schema.signature_record | 永久 | 电子签名记录 |

### 4.2 审计日志记录范围

| 模块 | 记录的操作 |
|------|-----------|
| 需求管理 | 创建、修改、删除、状态变更、评审、关联/取消关联 |
| 追溯管理 | 创建追溯、删除追溯 |
| 变更管理 | 创建变更、影响分析、审批、执行、验证 |
| 合规管理 | SOUP登记/更新/审查、基线锁定/解锁、安全分类 |
| 电子签名 | 签名意图创建、签名完成、签名验证 |
| 风险管理 | 创建/更新风险、分析、控制措施、状态变更 |
| 项目管理 | 项目创建/归档、成员变更、DCP评审 |
| 系统管理 | 用户创建/修改/禁用、角色分配、权限变更 |

### 4.3 审计日志不可篡改保障

| 保障层 | 机制 | 说明 |
|--------|------|------|
| 应用层 | 追加只写API | 审计日志仅提供INSERT接口，无UPDATE/DELETE |
| 数据库层 | 触发器 | BEFORE UPDATE/DELETE触发器抛出异常 |
| 数据层 | 哈希链 | 每条记录的current_hash依赖前一条记录的prev_hash |
| 校验层 | 定时校验 | 每日凌晨2:00运行哈希链完整性校验任务 |
| 告警层 | 篡改告警 | 哈希校验失败立即发送告警至运维 |

## 6. 通用业务规则

### 6.1 编号生成规则

> **C-02修订**：需求编号格式已对齐PRD定义，采用层级前缀+项目编号+序号。

| 实体 | 编号格式 | 示例 | 说明 |
|------|----------|------|------|
| 需求（URS） | `URS-{项目编号}-{NNNN}` | URS-PRJ01-0001 | 层级前缀+项目编号+4位序号 |
| 需求（PRS） | `PRS-{项目编号}-{NNNN}` | PRS-PRJ01-0001 | 同上 |
| 需求（SRS） | `SRS-{项目编号}-{NNNN}` | SRS-PRJ01-0001 | 同上 |
| 需求（DRS） | `DRS-{项目编号}-{NNNN}` | DRS-PRJ01-0001 | 同上 |
| 变更请求 | `CR-{YYYY}-{NNNN}` | CR-2026-0001 | 年份+4位序号，年度内自增 |
| 追溯链接 | `TL-{UUID8}` | TL-a1b2c3d4 | UUID前8位 |
| 风险项 | `RSK-{YYYY}-{NNNN}` | RSK-2026-0001 | 年份+4位序号 |
| 基线 | `BL-{YYYY}-{NNNN}` | BL-2026-0001 | 年份+4位序号 |
| 签名记录 | `SIG-{UUID8}` | SIG-e5f6g7h8 | UUID前8位 |
| 问题报告 | `PR-{YYYY}-{NNNN}` | PR-2026-0001 | 年份+4位序号，年度内自增 |

### 6.2 状态机规则

> **C-03修订**：状态机已补充层级特有状态，完整状态定义见02-核心业务流程设计 §1.1-1.2。

| 规则 | 说明 |
|------|------|
| 单向流转 | 需求状态只能按定义的方向流转，不可跳步 |
| 回退限制 | 仅 Rejected→Draft 和 Changed→Draft 允许回退，Closed/Retired/Baseline不可回退 |
| 层级特有状态 | PendingDecompose/Decomposed仅PRS/SRS适用；PendingVerify仅SRS/DRS适用 |
| 强制校验 | 状态变更前必须通过DCP门控校验和操作序列检查 |
| 事件触发 | 每次状态变更发布领域事件 |
| 审计记录 | 所有状态变更自动写入审计日志 |
| 并发控制 | 乐观锁防止并发状态变更 |

### 6.3 基线锁定规则

> **M-05修订**：基线锁定需双人签名，对比算法明确为SHA-256。

| 规则 | 说明 |
|------|------|
| 锁定条件 | 基线内所有需求状态为Approved/Verified/Baseline/Closed |
| 锁定操作 | 需QA经理（第一签）+ 第二签名人（研发经理或更高权限）双人电子签名 |
| 基线对比 | 锁定时使用SHA-256算法计算基线内容哈希，解锁后重新锁定时自动对比差异 |
| 锁定效果 | 基线内需求不可修改，需先解锁 |
| 解锁条件 | 需QA经理签名解锁，记录解锁原因 |
| 解锁审计 | 解锁操作写入审计日志，触发告警 |

## 8. 可用性与可靠性

### 8.1 可用性要求

| 指标 | 要求 |
|------|------|
| 系统可用性 | ≥99.5%（年停机≤43.8小时） |
| 计划维护窗口 | 每月1次，周六0:00-6:00 |
| 数据备份 | 每日全量备份 + 实时WAL归档 |
| RPO（恢复点目标） | ≤5分钟 |
| RTO（恢复时间目标） | ≤30分钟 |

### 8.2 容错机制

| 机制 | 说明 |
|------|------|
| 数据库主从 | PostgreSQL 主从复制，读操作可走从库 |
| Redis哨兵 | Redis Sentinel 高可用，主从自动切换 |
| 应用健康检查 | Spring Boot Actuator + Kubernetes liveness/readiness |
| 熔断器 | Resilience4j，下游服务异常时快速失败 |
| 降级策略 | 报表模块故障时，仪表盘显示缓存数据 |
| 死信队列 | 事件投递失败进入DLQ，人工介入 |

## 9. 安全约束

### 9.1 通信安全

| 措施 | 说明 |
|------|------|
| HTTPS | 全站HTTPS，TLS 1.2+ |
| HSTS | 严格传输安全头 |
| CSP | 内容安全策略，防止XSS |
| CORS | 白名单域名，限制跨域 |

### 9.2 数据安全

| 措施 | 说明 |
|------|------|
| 传输加密 | HTTPS + 字段级加密（签名密码） |
| 存储加密 | 密码bcrypt哈希，签名密码独立哈希 |
| 脱敏展示 | 列表接口手机号/邮箱部分隐藏 |
| SQL注入防护 | MyBatis-Plus参数化查询 |
| XSS防护 | 输入内容HTML转义，CSP头 |
| CSRF防护 | JWT无状态，无CSRF风险 |

### 9.3 访问控制

| 措施 | 说明 |
|------|------|
| IP白名单 | 管理后台仅内网访问 |
| 限流 | 按用户+API粒度，100次/分钟 |
| 会话管理 | JWT短期Token + Refresh Token |
| 异地登录检测 | 新设备/IP登录需二次验证 |
| 敏感操作确认 | 删除/签名等操作需二次确认 |

## 12. 部署与运维约束

### 12.1 部署架构

> **m-07修订**：部署架构与系统架构文档对齐，采用 Docker Compose 而非 Kubernetes。

| 组件 | 部署方式 | 说明 |
|------|----------|------|
| Med-RMS 后端 | Docker Compose | 单实例，滚动更新时手动切换 |
| Med-RMS 前端 | Nginx静态部署 | 与后端同机部署，反向代理 |
| PostgreSQL | Docker Compose 单实例 | WAL归档，每日全量备份 |
| Redis | Docker Compose 单实例 | 持久化AOF+RDB |
| Debezium | Docker Compose 独立容器 | 监听PostgreSQL WAL |
| MinIO | Docker Compose 单节点 | 开发/小规模部署足够 |
| Nginx | 同容器/独立容器 | 反向代理 + SSL终止 |

> **演进路线**：当单实例性能不足时，可升级为 Kubernetes + 主从数据库 + Redis Sentinel + MinIO 分布式。
