## 二、原型关键字段/流程摘要

### 2.1 changes 列表（MUST）
- 列表字段：变更编号、标题、**优先级（MAJOR/NORMAL/DOCUMENT/EMERGENCY）**、状态、类型、**影响范围**、申请人、申请日期
- 筛选：状态、优先级、关键字
- 侧栏：影响分析（URS/PRS/SRS/DRS/TC + 疑似/确认影响数 + 追溯链路 + 审批流程 5 步）

### 2.2 change-request（MUST，旧版）
- 卡片式列表：CR-ECG3-0012、CR-ECG3-0011
- 字段：CR 编号、类型（CORRECTIVE/ADAPTIVE/PERFECTIVE/EMERGENCY）、标题、关联需求、申请人、申请时间、**紧急程度**、**影响范围%**、状态
- 工作流 6 步：提交 → 影响评估 → 审批 → 执行 → 验证 → 关闭
- 影响评估：受影响需求、关联测试用例、涉及法规条款、**影响范围 %**
- 审批历史：头像+姓名+时间+动作+结果

### 2.3 change-create（MUST）
- 字段：变更编号（auto）、**变更类型**、**优先级**、**影响范围**、变更标题、变更描述、变更理由、影响分析

### 2.4 change-edit（COULD）
- 字段：CRQ-2026-0042、变更类型（功能性/非功能性/法规/安全）、**优先级 MUST/SHOULD/COULD**、影响范围（单/多/跨）、标题、描述、理由、影响分析

### 2.5 change-approve（COULD）
- 卡片展示：CRQ 编号、类型、优先级、影响范围、申请人
- 详情：描述、理由、影响分析、关联需求
- **审批操作：审批意见（必填）+ 电子签名（21 CFR Part 11 提示）**
- 操作：拒绝 / 批准并签署

### 2.6 change-execute（COULD）
- 流程时间线：申请 → 影响分析 → 审批 → 执行 → 验证
- 警告横幅：执行前请确认
- **执行清单表**：任务、负责人、状态（已完成/进行中/待执行）、完成进度%、计划日期
- 执行确认表单：执行结论、相关文档链接

### 2.7 change-verify（COULD）
- 流程时间线（5 步）
- **验证清单**：复选框 + 负责人/计划/实际 + 结果（通过/失败/待验证）
- **上传验证证据**（拖拽上传，PDF/Word/Excel/图片，≤20MB）
- 验证结论下拉：全部通过/部分通过/验证失败
- 验证说明、备注

### 2.8 change-impact-analysis（MUST）
- 步骤指示器：填写信息 → **影响分析** → 影响评估 → 提交审批
- **分析范围配置**：分析方向（仅下游/双向/仅上游）、**分析深度（一级/二级/全链路）**
- 4 个分析选项卡片：需求、追溯链、风险、测试用例
- 变更目标对象（URS-P-003 等）
- 上游依赖对象
- **分析选项**：变更紧迫度（普通/紧急/关键）、是否包含历史版本
- 启动分析按钮

### 2.9 change-impact-result（MUST）
- 汇总统计：高/中/低影响项 + 受影响需求
- 三段式列表：高/中/低影响项，**含影响等级点 + 查看详情按钮 + 复选框**
- 底部：已选择 N 项 + 提交影响评估报告

### 2.10 change-approvals（COULD）
- 工作台顶部 4 个统计卡：待审批/已通过/已拒绝/平均处理时长
- 筛选：状态、变更类型（纠正性/适应性/完善性/紧急）、关键字
- 审批卡：标题+ID+类型徽标+状态+meta+描述+批准/拒绝/查看详情按钮

## 四、偏差清单

### 偏差 1（BLOCKER）：缺失独立变更执行页面
- **原型**：`change-execute-原型.html`
- **现状**：路由中无 `/changes/:id/execute`，ChangeList 仅用 `ElMessageBox.confirm` 弹出确认后即 `changeApi.execute(id, {})`
- **缺失**：
  - 流程时间线（5 步：申请/分析/审批/执行/验证）
  - 警告横幅"执行前请确认"
  - **执行清单表**（任务、负责人、状态、完成进度%、计划日期）—— 这是原型核心模块
  - 执行结论（必填 textarea）
  - 相关文档链接
  - 保存草稿 / 确认执行完成 按钮
- **影响**：执行阶段无法落地"分项任务推进"，无法记录每项完成人/进度/计划日期，违反原型 MUST 级流程

### 偏差 2（BLOCKER）：缺失独立变更验证页面
- **原型**：`change-verify-原型.html`
- **现状**：路由中无 `/changes/:id/verify`，ChangeList 用 ElMessageBox + `changeApi.verify(id)` 一句话完成
- **缺失**：
  - 5 步流程时间线
  - **验证清单**（复选框 + 负责人/计划/实际 + 通过/失败/待验证标签）
  - **证据上传**（拖拽上传，PDF/Word/Excel/图片 ≤20MB，差异对照表等多文件管理）
  - 验证结论下拉（全部通过/部分通过/验证失败）
  - 验证说明 + 备注
  - "全部通过才能关闭"业务规则
- **影响**：QA 验证无据可查，证据链缺失，违反 21 CFR Part 11 审计要求

### 偏差 3（MAJOR）：缺失独立"我的审批"工作台
- **原型**：`change-approvals-原型.html`
- **现状**：路由无 `/changes/approvals`，无对应页面
- **缺失**：
  - 4 个统计卡：待审批/已通过/已拒绝/平均处理时长
  - 筛选（状态 + 变更类型 + 关键字）
  - 审批卡片列表（标题+ID+类型徽标+meta+描述+批准/拒绝/查看详情）
  - 类型徽标：纠正性/适应性/完善性/紧急
- **影响**：研发总监、QA 负责人没有"待我审批"集中入口，与原型 COULD 反馈要求的"补回审批工作台"相悖

### 偏差 4（MAJOR）：变更优先级字段体系不统一
- **原型要求**：
  - `changes-原型.html`：MAJOR / NORMAL / DOCUMENT / EMERGENCY
  - `change-edit-原型.html`：MUST / SHOULD / COULD
- **现状**：ChangeList 仅展示 `urgency`（NORMAL/URGENT），无 priority 字段；ChangeRequest 仅展示 `priority`（HIGH/MEDIUM/LOW 映射），无原文 MUST/SHOULD/COULD；ChangeCreate 表单中无 priority 字段
- **缺失**：
  - ChangeList 缺少独立的"优先级"列（按原型必须用徽标显示 MAJOR/NORMAL/DOCUMENT/EMERGENCY）
  - ChangeCreate 缺少优先级选择器
  - 缺少与原型"编辑"页中 MUST/SHOULD/COULD 三档可选
- **影响**：与原型 v1.5 中"优先级 MUST/SHOULD/COULD"明确值不符，且列表无法按优先级筛选

### 偏差 5（MAJOR）：影响范围字段缺失
- **原型要求**：
  - `change-edit`：影响范围下拉 = 单系统 / 多系统 / 跨模块
  - `change-request`：影响范围 %（带颜色条 low/medium/high）
  - `change-impact-result`：影响等级 dot（高/中/低）
- **现状**：
  - ChangeCreate 表单无"影响范围"字段
  - ChangeList "影响范围"列只展示 URS/PRS/SRS 三个 tag（缺 DRS/TC），无百分比进度条
  - ChangeRequest 显示 `impactScope` 字段但 ChangeCreate 未提供填写入口
- **影响**：与 edit/approve 原型不一致；影响等级无法在列表页直观判断

### 偏差 6（MAJOR）：5/6 步生命周期流程条未可视化
- **原型要求**：
  - `change-request`：工作流 6 步（提交/影响评估/审批/执行/验证/关闭）
  - `change-execute`：流程时间线（5 步）
  - `change-verify`：流程时间线（5 步）
- **现状**：
  - ChangeList 无流程条，仅标签显示状态
  - ChangeRequest 有 `el-timeline` 审批流程，但只显示已发生的审批节点，无整体进度
  - ChangeCreate 底部 Alert 用 `<ol>` 文字列出 5 步，**非可视化步骤条**
- **影响**：用户难以直观判断当前变更处于生命周期哪一阶段

### 偏差 7（MAJOR）：ChangeRequest 详情页与 change-approve 原型差距大
- **原型要求**（change-approve）：卡片式详情 + 完整描述/理由/影响分析/关联需求 + **审批意见（必填）** + **电子签名（21 CFR Part 11 提示）** + "批准并签署"按钮
- **现状**：
  - ChangeRequest 用 el-descriptions 表格形式，缺少 change-approve 中的"详情卡片"组织
  - "审批"按钮打开 ChangeRequest 内嵌的 dialog，仅有"决策（批准/拒绝）+ 意见"两字段
  - **无电子签名说明文案**（"根据 21 CFR Part 11，您的电子签名将成为本审批的法定签名"）
  - **无"批准并签署"按钮文案**
  - ChangeList 中 `showApproveDialog` 复用同一简化版 dialog
- **影响**：21 CFR Part 11 合规要求未在前端落地，电子签名流程无 UI 引导

### 偏差 8（MINOR）：ChangeList 列表与 changes/change-request 原型差距大
- **原型要求**：
  - `changes`：表格 8 列（编号/标题/优先级/状态/类型/影响范围/申请人/日期）+ 状态/优先级/关键字筛选
  - `change-request`：卡片式 2 列网格 + 影响范围 % 条 + 工作流进度条
- **现状**：ChangeList 用 el-table，列有变更编号/类型/原因/影响范围/紧急程度/状态/申请时间/操作，**无独立的"标题"列、无"申请人"列、无"优先级"列**；筛选只有"状态 + 需求编号"两种，**无"优先级"筛选、无"关键字全文搜索"**
- **影响**：列表信息密度低于原型，缺少常用筛选维度

### 偏差 9（MINOR）：ChangeImpactAnalysis 与原型差异
- **原型要求**（change-impact-analysis）：
  - 步骤指示器（填写信息/影响分析/影响评估/提交审批）
  - **分析方向**（仅下游/双向/仅上游）
  - **分析深度**（一级/二级/全链路）
  - 变更目标对象 + 上游依赖对象
  - **变更紧迫度** + **是否包含历史版本**
  - "启动影响分析"按钮（带 loading 动画 1.5s 后跳转到 result）
- **原型要求**（change-impact-result）：
  - 4 个汇总统计（高/中/低 + 受影响需求）
  - **三段式分组**（高/中/低影响项）每项含复选框 + 影响等级 dot + 查看详情按钮
  - 底部 "提交影响评估报告" 按钮
- **现状**：
  - ChangeImpactAnalysis 实现了统计卡 + 影响项表格，**但缺少步骤指示器**
  - **缺少分析方向、分析深度字段**
  - **缺少变更紧迫度/历史版本字段**
  - 4 个分析选项与原型不同：原型是"需求/追溯链/风险/测试用例"，实现是"需求追溯/测试用例/基线影响/SOUP组件"
  - **未实现三段式分组**（高/中/低）
  - **缺少"提交影响评估报告"按钮**（仅"重新分析"）
  - 无"启动分析"loading 动画与跳页
- **影响**：分析维度比原型少，缺少报告提交动作；选项定义不一致需业务确认

### 偏差 10（MINOR）：change-edit 独立路由缺失
- **原型要求**：`/changes/:id/edit`，ChangeRequestController.update
- **现状**：
  - 路由无 `/changes/:id/edit`
  - ChangeRequest 有"编辑"按钮 + `showEditDialog`，但 dialog **只含变更标题/描述/理由 3 字段**
  - 原型 edit 表单含变更类型、**优先级**、**影响范围**、标题、描述、理由、影响分析 7+ 字段
- **影响**：编辑能力不完整，无法修改类型/优先级/影响范围

### 偏差 11（MINOR）：疑似/确认影响（suspect）标记缺失
- **原型要求**（changes）：影响链路节点带 "确认影响/疑似影响" 标签 + suspect 背景
- **现状**：ChangeRequest 影响项表只有 `impactLevel`（HIGH/MEDIUM/LOW），**无 suspect/affected 二元状态标记**
- **影响**：无法区分"已确认受影响"与"疑似受影响"两类节点

### 偏差 12（MINOR）：优先级筛选缺失
- **原型要求**（changes）：状态 + 优先级 + 关键字 三合一筛选
- **现状**：ChangeList 只有"状态 + 需求编号"，**无优先级筛选、无关键字全文搜索**
- **影响**：与原型列表顶部工具栏不符

### 偏差 13（MINOR）：change-impact-result 高/中/低影响项复选 + 操作按钮缺失
- **原型要求**：每条影响项前有 `impact-checkbox`，分组底部有"已选择 N 项" + "提交影响评估报告"
- **现状**：ChangeImpactAnalysis 影响项表无复选框、无选择计数器、无"提交影响评估报告"按钮
- **影响**：无法批量勾选进入下一阶段

### 偏差 14（MINOR）：状态机取值范围偏差
- **原型要求**（changes 列表 statusMap）：submitted / assessing / approved / implemented / closed
- **原型要求**（change-approve）：待审批
- **原型要求**（change-execute/verify 流程时间线）：已完成 / 进行中 / 待执行 / 待验证
- **现状**（ChangeRequest.getStatusLabel）：DRAFT / PENDING_ANALYSIS / PENDING_APPROVAL / APPROVED / REJECTED / IN_PROGRESS / COMPLETED
- **现状**（ChangeList.changeStatusLabels）：DRAFT / ANALYZING / PENDING_APPROVAL / APPROVED / EXECUTING / VERIFIED / CLOSED
- **问题**：两个前端文件的状态枚举不一致（ChangeList 有 EXECUTING/VERIFIED，ChangeRequest 有 PENDING_ANALYSIS/IN_PROGRESS/COMPLETED）；缺少原型 `assessing` 之外的 `implementing`、`verified` 等过渡态
- **影响**：状态机未与原型对齐，后端返回的状态可能在前端无对应样式

### 偏差 15（TRIVIAL）：视觉表现差距
- 原型 `changes-原型.html` 用 row hover 高亮（`background: #f5f9ff`），ChangeList 用 el-table 默认 stripe
- 原型优先级使用彩色徽标（MAJOR 红 / NORMAL 黄 / DOCUMENT 蓝 / EMERGENCY 红描边），ChangeList 用 el-tag 默认色
- 原型 `change-request-原型.html` 用卡片式 2 列网格 + 影响范围 % 进度条（low 绿/medium 黄/high 红），ChangeList 用 el-table
- 影响：交互体验感观与原型差距较大

### 偏差 16（TRIVIAL）：紧急程度枚举值不一
- `change-create` 与 `change-request` 原型中"紧急程度"：普通/紧急
- `change-approvals` 原型中"紧急程度"：高/中/低/紧急
- `change-impact-analysis` 原型中"变更紧迫度"：普通(5d)/紧急(2d)/关键(立即)
- 现状 ChangeCreate/ChangeList：NORMAL / URGENT（缺关键 critical 一档）
- 影响：枚举不统一，业务侧口径需确认

### 偏差 17（TRIVIAL）：change-execute/verify 表单字段映射缺失
- 原型 execute 含"执行结论/相关文档链接"
- 原型 verify 含"验证结论(下拉)/验证说明/备注/证据上传"
- 现状 ChangeList 中 `changeApi.execute(id, {})` `changeApi.verify(id)` 均无 body
- 影响：后端 API 即使支持也未传参

### 偏差 18（TRIVIAL）：我的审批无菜单入口
- 路由完全缺失 `/changes/approvals`
- 侧边栏（Layout.vue 通用）即便加上"我的审批"菜单也无法跳转
- 影响：研发总监/QA 主管无统一入口
