## 1. 系统级枚举（dict_type.category = 'system'）

### 1.1 REQ_TYPE — 需求类型

| 编码 | 标签 | 说明 | 适用场景 |
|------|------|------|----------|
| URS | 用户需求规格 | L1层级，来源于用户/市场/法规 | 顶层需求，由用户需求拆解为PRS |
| PRS | 产品需求规格 | L2层级，产品视角的功能需求 | 由URS拆解而来，拆解为SRS |
| SRS | 软件需求规格 | L3层级，软件实现视角的需求 | 由PRS拆解而来，拆解为DRS |
| DRS | 设计需求规格 | L4层级，设计实现细节需求 | 由SRS拆解而来，最底层 |

### 1.2 REQ_LEVEL — 需求层级

| 编码 | 标签 | 对应REQ_TYPE | 说明 |
|------|------|-------------|------|
| L1 | 用户需求 | URS | 最高层级，源自用户/法规 |
| L2 | 产品需求 | PRS | 产品视角功能描述 |
| L3 | 系统需求 | SRS | 软件实现规格 |
| L4 | 设计需求 | DRS | 设计实现细节 |

### 1.3 REQ_STATUS — 需求状态

| 编码 | 标签 | 适用层级 | 前置状态 | 允许操作 | 说明 |
|------|------|---------|----------|----------|------|
| Draft | 草稿 | 全部 | — | 编辑、删除、提交 | 初始状态 |
| PendingDecompose | 待拆解 | PRS/SRS | Approved | 创建下级需求、建立追溯 | PRS/SRS评审通过后需拆解 |
| Decomposed | 已拆解 | PRS/SRS | PendingDecompose | 编辑追溯、标记实现 | 下级需求全部创建 |
| Submitted | 已提交 | 全部 | Draft | 取消提交 | 等待分配评审人 |
| InReview | 评审中 | 全部 | Submitted | 评审操作 | 评审人正在评审 |
| Approved | 已批准 | 全部 | InReview | URS/DRS→标记实现；PRS/SRS→待拆解 | 评审通过 |
| Rejected | 已驳回 | 全部 | InReview | 修改后重新提交 | 评审不通过 |
| PendingVerify | 待验证 | SRS/DRS | Implemented | 上传验证证据 | 需要验证证据 |
| Implemented | 已实现 | 全部 | Approved/Decomposed | SRS/DRS→提交验证 | 标记为已开发 |
| Verified | 已验证 | SRS/DRS | PendingVerify | 加入基线 | 验证证据通过 |
| Baseline | 已基线 | 全部 | Verified/Approved | — | 已纳入基线，锁定 |
| Changed | 已变更 | 全部 | Baseline | — | 基线后发生变更 |
| Closed | 已关闭 | 全部 | Verified/Baseline | 重新打开 | 需求完结 |
| Retired | 已退役 | 全部 | Closed | — | 需求废弃 |

| 编码 | 标签 | 适用场景 | 说明 |

| review | 审核 | 评审审核 | 已审核内容 |
| release | 发布 | 基线发布、报告发布 | 授权发布 |

### 1.15 MEANING_CODE — 签名角色码

| 编码 | 标签 | 适用场景 | 说明 |
|------|------|----------|------|
| author | 编制人 | 需求编制、文档编写 | 创建内容的签名 |
| reviewer | 审核人 | 评审审核 | 审核内容的签名 |
| approver | 批准人 | 变更审批、基线锁定 | 最终批准的签名 |
| second_approver | 第二批准人 | 基线双人签名 | 基线锁定的第二签名人 |

> **枚举澄清**：`SIGN_INTENT`（签名意图）和 `MEANING_CODE`（签名角色码）是两个独立维度：
> - SIGN_INTENT 描述签名的**业务目的**（approve/confirm/review/release），对应21 CFR Part 11 §11.50的签名含义声明，存储在 signature_record.sign_intent
> - MEANING_CODE 描述签名人的**业务角色**（author/reviewer/approver/second_approver），用于权限校验和审计追踪，存储在 signature_record.meaning_code
> - 两者关系：一个角色可以执行不同意图。例如 approver 角色可以执行 approve 意图（审批通过）或 review 意图（审核确认）

## 3. 业务编码规则

### 3.1 需求编号

```
格式：{层级前缀}-{项目编号}-{序号}
示例：URS-PRJ001-0001

层级前缀：URS | PRS | SRS | DRS
项目编号：项目code字段（大写字母+数字，最长10位）
序号：4位数字，从0001开始，每个项目+层级独立编号

正则：^(URS|PRS|SRS|DRS)-[A-Z0-9]{1,10}-\d{4}$
```

### 3.2 变更请求编号

```
格式：CR-{项目编号}-{序号}
示例：CR-PRJ001-0001

正则：^CR-[A-Z0-9]{1,10}-\d{4}$
```

### 3.3 问题报告编号

```
格式：PR-{项目编号}-{序号}
示例：PR-PRJ001-0001

正则：^PR-[A-Z0-9]{1,10}-\d{4}$
```

### 3.4 测试用例编号

```
格式：TC-{项目编号}-{序号}
示例：TC-PRJ001-0001

正则：^TC-[A-Z0-9]{1,10}-\d{4}$
```

### 3.5 基线编号

```
格式：BL-{项目编号}-{序号}
示例：BL-PRJ001-0001

正则：^BL-[A-Z0-9]{1,10}-\d{4}$
```
