## 二、严重问题（Critical）—— 必须修订

### C-01：requirement 主表缺少系统架构定义的字段

**位置**：02-DDL/med_rms_ddl.sql §4.1.1 `req_schema.requirement`

**问题**：系统架构 v1.1 §6.1.1 `t_requirement` 定义了以下字段，但 DDL 中完全缺失：

| 系统架构字段 | 说明 | DDL状态 |
|-------------|------|---------|
| `safety_class` | 软件安全分类（A/B/C） | ❌ 缺失 |
| `requirement_category` | 需求分类（SOFTWARE/HARDWARE/BOTH） | ❌ 缺失 |
| `baseline_id` | 关联基线（基线化后填写） | ❌ 缺失 |
| `is_deleted` | 软删除标记 | ❌ 缺失 |
| `current_version` | 当前版本号 | ❌ 缺失 |
| `module_id` | 所属模块 | ❌ 缺失 |
| `updated_by` | 最后修改人 | ❌ 缺失 |

**影响**：需求管理是核心模块，缺少这些字段将导致：
- 无法满足 IEC 62304 软件安全分类追溯要求
- 基线关联无法实现
- 软删除无法支持，审计追踪不完整

**修订建议**：在 `req_schema.requirement` 表补充上述字段。

---

### C-02：audit_log 表 prev_hash 类型与系统架构不一致

**位置**：02-DDL/med_rms_ddl.sql §4.5.3

**问题**：系统架构 v1.1 §6.3.1 定义 `prev_hash CHAR(64) NOT NULL`，但 DDL 定义为 `TEXT`：

```sql
-- DDL 定义（错误）
prev_hash       TEXT,

-- 系统架构要求
prev_hash CHAR(64) NOT NULL
```

**影响**：SHA-256 哈希值固定 64 字符，使用 TEXT 会导致：
- 存储空间浪费
- 无法通过数据库约束保证长度一致
- 哈希链校验逻辑可能出现边界问题

**修订建议**：将 `prev_hash` 字段类型改为 `CHAR(64)`。

---

### C-03：DDL 缺少 PostgreSQL 触发器实现

**位置**：02-DDL/med_rms_ddl.sql §4.5.3

**问题**：系统架构 v1.1 §6.3.1 明确要求数据库层触发器阻止审计日志 UPDATE/DELETE 操作：

```sql
-- 系统架构定义的触发器
CREATE TRIGGER trg_prevent_audit_update
    BEFORE UPDATE ON t_audit_log
    FOR EACH ROW EXECUTE FUNCTION fn_prevent_audit_update();

CREATE TRIGGER trg_prevent_audit_delete
    BEFORE DELETE ON t_audit_log
    FOR EACH ROW EXECUTE FUNCTION fn_prevent_audit_delete();
```

但 DDL 中完全未实现这些触发器。

**影响**：21 CFR Part 11 合规性缺失，应用层保护可被 DBA 直接操作绕过。

**修订建议**：在 DDL 中补充审计日志保护触发器。

---

### C-04：JPA 实体类缺少 Outbox 表定义

**位置**：01-详细设计/各模块详细设计.md

**问题**：系统架构 ADR-010 要求使用事务性发件箱模式（Transactional Outbox + Debezium CDC），DDL 中已定义 `esign_schema.outbox` 表，但所有详细设计文档的 JPA 实体类图中均未出现 `Outbox` 实体。

**影响**：
- 跨模块事件驱动机制缺少对应的 Java 实体
- Debezium CDC 监听 Outbox 表但应用层无对应实体
- 开发人员可能忽略 Outbox 表的使用

**修订建议**：在支撑域详细设计中补充 Outbox 实体定义。

## 三、重要问题（Major）—— 建议修订

### M-01：技术栈标注与实际实现不一致

**位置**：01-详细设计/req-mgr-详细设计.md §1

**问题**：详细设计文档标注技术栈为 **"JPA/Hibernate"**，但：
- DDL 采用原生 SQL 定义（MyBatis-Plus 原型）
- 系统架构 §4.1 选择 ORM 为 **MyBatis-Plus 3.5.x**
- 其他模块详细设计也未使用 JPA 注解风格

**影响**：技术选型标注不清，可能导致开发团队困惑。

**修订建议**：明确技术栈为 MyBatis-Plus，补充 Mapper 接口定义替代 JPA Repository。

---

### M-02：状态机转换规则与概要设计存在偏差

**位置**：01-详细设计/req-mgr-详细设计.md §5.1

**问题**：详细设计定义 `APPROVED → PENDING_VERIFY` 转换（规则8），但：

- 02-核心业务流程设计 §1.1 状态机定义：`IMPLEMENTED → PENDING_VERIFY`
- 系统架构无此直接转换

**差异**：

| 转换 | 详细设计 | 概要设计/系统架构 |
|------|---------|------------------|
| → PENDING_VERIFY | APPROVED 可直接转换 | 需先到 IMPLEMENTED |

**影响**：需求状态转换逻辑不正确，影响业务流程实现。

**修订建议**：统一状态机转换规则为：只有 `IMPLEMENTED` 状态可转换到 `PENDING_VERIFY`。

---

### M-03：JWT Blacklist 表定义在详细设计中缺失

**位置**：01-详细设计/e-sign-详细设计.md

**问题**：e-sign 详细设计类图中包含 `JwtBlacklist` 实体，但 req-mgr、trace-mgr、chg-mgr、compliance 等模块的详细设计中未体现 JWT 黑名单的使用逻辑（特别是登出时写入黑名单）。

**影响**：JWT 黑名单是 ADR-007 的核心要求，所有模块的登出操作都需要调用黑名单机制，但其他模块详细设计未体现此协作。

**修订建议**：在各模块详细设计中补充登出时调用 JWT 黑名单的时序。

---

### M-04：详细设计缺少 problem_report API 端点

**位置**：03-OpenAPI/med-rms-openapi.yaml、02-核心业务流程设计

**问题**：
- 概要设计评审报告 M-04 指出"问题报告管理完全缺失"
- compliance 详细设计补充了 ProblemReport 实体和流程
- 但 OpenAPI 规范中未找到问题报告相关的 API 定义

**影响**：问题报告是 ISO 13485 纠正预防措施的核心功能，缺少 API 将导致无法实现问题报告的增删改查。

**修订建议**：在 OpenAPI 规范中补充 `/problem-reports` 等 API 端点。

## 六、评审结论

### 总体评价

详细设计文档体系完整、结构清晰，15 份交付物覆盖了所有业务模块的类图、时序图、伪代码，DDL 和 OpenAPI 规范基本齐全。设计深度足够，可直接指导开发。

### 核心风险

1. **数据模型层是最大风险点** — requirement 主表缺少 7 个字段，如果不修订将导致合规性缺失（IEC 62304 追溯链不完整）
2. **技术栈标注与实现不符** — JPA vs MyBatis-Plus 混用可能造成团队困惑
3. **Outbox 实体缺失** — 影响跨模块事件驱动机制的实现
