## 1. 产品目标

本系统服务于武汉中旗生物医疗电子有限公司内部，解决医疗器械需求管理中合规追溯难、变更影响不可控、层级转化无标准三大核心痛点。

| 目标编号 | 目标 | 衡量方式 | 目标值 |
|---------|------|---------|-------|
| G1 | **合规零缺陷**：确保每次NMPA/ISO 13485审计中，需求管理相关零不符合项 | 审计不符合项数 | = 0 |
| G2 | **追溯全链路**：实现URS→PRS→SRS→DRS四层需求纵向100%可追溯 | 需求全链路闭环率 | ≥ 85% |
| G3 | **变更零失控**：任何需求变更的影响范围可自动评估、受影响项可自动标记、追溯链可自动更新 | 变更影响评估覆盖率 | 100% |

**目标正交性说明**：G1关注合规结果，G2关注数据完整性，G3关注过程可控性——三者互不包含、缺一不可。

## 2. 用户故事

### US-1 研发总监：全局审批监督
> 作为研发总监，我需要在仪表盘上一目了然地看到所有需求的状态分布、合规达标率和追溯完整率，以便我在DCP阶段门评审时快速判断项目是否具备进入下一阶段的条件。

**验收标准**：
- Given 系统中存在处于DCP门的项目 / When 研发总监打开仪表盘 / Then 可看到该项目需求合规率、追溯闭环率、高风险需求数的汇总卡片，且数据实时更新（延迟≤5分钟）

### US-2 项目经理：变更影响全景掌控
> 作为项目经理，我需要当任何需求发生变更时，系统自动评估并展示所有受影响的下游需求、设计文档和测试用例，以便我准确评估变更代价并组织评审。

**验收标准**：
- Given URS-001发生变更 / When 项目经理打开变更影响评估面板 / Then 系统自动列出所有下游PRS/SRS/DRS及关联测试用例，标记为"suspect"状态，并计算影响范围百分比

### US-3 系统架构师：L2→L3高效转化
> 作为系统架构师，我需要在可视化拆解工作台中，将一个PRS逐条拆解为多个SRS，并自动继承上游的法规关联和追溯关系，以便我高效完成需求层级转化而不遗漏合规约束。

**验收标准**：
- Given PRS-010已通过评审 / When 系统架构师在拆解工作台中创建SRS / Then 系统自动继承PRS-010的法规条款关联，并在拆解完成后校验所有SRS是否覆盖PRS的全部验收标准

### US-4 合规专员：合规证据一键生成
> 作为合规专员，我需要在项目DCP评审前，一键生成包含完整追溯矩阵、电子签名记录和审计日志的DHF合规证据包，以便我高效应对审计而不用手工拼凑文档。

**验收标准**：
- Given 项目完成某阶段所有需求评审 / When 合规专员点击"生成证据包" / Then 系统在≤30秒内生成包含追溯矩阵、评审记录、电子签名日志的PDF证据包，且追溯闭环率≥95%

### US-5 产品经理：L1/L2需求高效生产
> 作为产品经理，我需要在需求收集池中汇总来自客户、法规、市场的多渠道需求，并将其转化为结构化的URS和PRS，以便我系统化地管理需求来源和优先级。

**验收标准**：
- Given 多条需求进入收集池 / When 产品经理选择并转化为URS / Then 系统自动记录需求来源、原始描述，并引导填写URS必填字段（法规关联、优先级、验收标准）

### US-6 研发工程师：L4需求落地执行
> 作为研发工程师，我需要从SRS直接接收拆解后的DRS，并在完成实现后标记对应测试用例为"待验证"，以便需求实现和验证之间的衔接不断裂。

**验收标准**：
- Given SRS-020已拆解为DRS / When 研发工程师标记DRS-020-01为"已实现" / Then 系统自动关联的测试用例状态更新为"待验证"，并通知测试工程师

### US-7 测试工程师：需求-用例覆盖验证
> 作为测试工程师，我需要在创建测试用例时直接关联到SRS/DRS，并实时查看需求覆盖率，以便我确保每个需求都有对应的验证用例且无遗漏。

**验收标准**：
- Given SRS-020处于"已评审"状态 / When 测试工程师创建测试用例并关联 / Then 系统实时更新SRS-020的测试覆盖率，若覆盖率<100%则在该需求卡片上显示警告标识

### US-8 质量工程师：全层质量监督
> 作为质量工程师，我需要在任何需求评审未通过或追溯链断裂时收到系统通知，以便我及时介入跟踪整改，确保质量体系持续有效。

**验收标准**：
- Given 任意需求评审结果为"不通过"或追溯链检测到断裂 / When 事件发生 / Then 质量工程师在≤1分钟内收到系统通知（含需求编号、问题描述、建议操作）

## 3. 用户研究洞察

基于用户研究员瑞思的深度调研，提炼5条关键洞察，每条洞察均直接驱动PRD中的功能决策：

| 洞察编号 | 洞察内容 | PRD功能映射 |
|---------|---------|------------|
| INS-1 | **四层架构最大价值不在"存储"而在"转化"**——拆解工作台是核心 | 7.1节定义层级转化规则；7.2节设计拆解工作台为独立功能模块 |
| INS-2 | **合规专员是"隐形核心用户"**——合规驱动是系统成功的底层逻辑 | 7.5节合规管理提升为核心模块而非附加；G1目标定义 |
| INS-3 | **变更管理是8类用户共同的"最大焦虑点"**——应提升为跨模块枢纽功能 | 7.4节变更管理独立为顶层模块，与需求、追溯、合规双向联动 |
| INS-4 | **角色间信息不对称是最大效率瓶颈**——系统应成为"信息同步器" | 7.9节通知机制设计多角色订阅；7.8节仪表盘多视角视图 |
| INS-5 | **项目管理分两步走**——先"需求-任务联动"，再"完整项目管理" | 第10节分阶段交付计划：P1需求管理核心→P2项目管理扩展 |

## 7. 功能模块详细设计

### 7.1 需求层级架构

#### 7.1.1 四层架构总览

```
┌─────────────────────────────────────────────────────────┐
│ L1: URS（用户需求规格）                                    │
│ 来源：用户/客户/市场/法规                                   │
│ ↓ N:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L2: PRS（产品需求规格）                                    │
│ 来源：URS拆解+产品定义                                     │
│ ↓ 1:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L3: SRS（系统需求规格）                                    │
│ 来源：PRS拆解+系统架构约束                                  │
│ ↓ 1:1→1:N 转化                                           │
├─────────────────────────────────────────────────────────┤
│ L4: DRS（设计需求规格）                                    │
│ 来源：SRS拆解+设计约束                                     │
│ → 关联：测试用例、设计文档、代码模块                          │
└─────────────────────────────────────────────────────────┘
```

#### 7.1.2 L1: URS — 用户需求规格

| 属性 | 定义 |
|------|------|
| **核心定义** | 从用户/客户/法规视角描述的需求，回答"用户需要什么" |
| **边界** | 不涉及技术实现方案，仅描述用户可感知的行为或约束 |
| **主要生产者** | 产品经理 |
| **主要消费者** | 产品经理（评审）、合规专员（合规审查）、系统架构师（下游转化输入） |
| **IPD阶段门映射** | DCP1（概念阶段）输入 |

**必填字段清单**：

| 字段名 | 类型 | 说明 | 校验规则 |
|--------|------|------|---------|
| URS-ID | 自动生成 | 格式：URS-{项目编号}-{序号} | 系统自动，不可修改 |

#### 7.1.6 层级间转化规则

| 转化路径 | 基数 | 转化条件 | 转化时自动执行 | 校验规则 |
|---------|------|---------|-------------|---------|
| URS→PRS | 1:N | URS状态为"已通过" | 法规关联继承、来源追溯建立 | 每条URS至少拆解为1条PRS；所有PRS验收标准覆盖URS验收标准 |
| PRS→SRS | 1:N | PRS状态为"已通过" | 法规关联继承、接口需求传递 | 每条PRS至少拆解为1条SRS；所有SRS验收标准覆盖PRS验收标准 |
| SRS→DRS | 1:N | SRS状态为"已通过" | 法规关联继承、接口规格传递、测试策略继承 | 每条SRS至少拆解为1条DRS；所有DRS验收标准覆盖SRS验收标准 |

**转化完整性校验**（拆解工作台内置）：
1. **覆盖率校验**：下游所有验收标准的并集是否覆盖上游全部验收标准
2. **法规一致性校验**：下游的法规关联是否包含上游的全部法规关联
3. **完整性校验**：所有下游需求的必填字段是否均已填写

**反向追溯（N:1汇总）**：
- 允许多条URS汇总为1条PRS（当多个用户需求指向同一产品能力时）
- 允许多条PRS汇总为1条SRS（当多个产品需求通过同一系统方案实现时）
- 系统记录每条追溯关系的创建时间、创建人、追溯原因

#### 7.1.7 IPD阶段门映射

| DCP | 阶段 | 需求层级行为 | 系统校验规则 |
|-----|------|------------|------------|
| DCP1 | 概念 | URS创建与评审 | 所有URS必须"已通过"；每条URS至少关联1条法规条款 |
| DCP2 | 计划 | URS→PRS转化，PRS评审 | 所有URS已拆解为PRS；所有PRS必须"已通过"；追溯完整率≥90% |
| DCP3 | 开发 | PRS→SRS→DRS转化，实现 | 所有PRS已拆解为SRS；所有SRS已拆解为DRS；DRS实现完成率≥80% |
| DCP4 | 验证 | 测试验证 | DRS验证通过率≥95%；测试覆盖率100%；追溯闭环率≥85% |
| DCP5 | 发布 | 全链路确认 | 全链路追溯闭环率≥95%；合规证据包可生成；无高风险未闭环项 |

### 7.2 需求全生命周期管理

#### 7.2.1 需求创建与编辑（FR-0.6）

**功能描述**：支持四层需求的创建、编辑，按层级自动校验必填字段。

**验收标准**：
- Given 用户选择创建某层级需求 / When 填写表单并提交 / Then 系统校验所有必填字段，不完整则提示具体缺失字段，完整则保存为"草稿"状态
- Given 需求处于"草稿"状态 / When 用户编辑后保存 / Then 系统重新校验必填字段并记录修改审计日志
- Given 任意层级需求被创建 / When 保存成功 / Then 系统自动生成该层级的ID（格式：{层级缩写}-{项目编号}-{3位序号}）

#### 7.2.2 需求状态机流转（FR-0.1）

**功能描述**：每层需求按7.1节定义的状态机流转，所有状态变更记录在审计日志中。

**验收标准**：
- Given 需求当前状态为S1 / When 用户执行操作O / Then 需求状态变更为S2，审计日志记录变更人、时间、原状态、新状态
- Given 需求当前状态为S1 / When 用户尝试执行不允许的操作O' / Then 系统拒绝操作并提示"当前状态不允许此操作"
- Given 需求已被基线化 / When 发起变更 / Then 状态变为"已变更"，所有下游关联项自动标记为"suspect"（见7.4节）

#### 7.2.3 需求评审在线化（FR-0.4）

**功能描述**：评审发起→评审执行→评审结论全在线，评审意见结构化可追溯。

**评审流程**：

```
发起评审 → 系统通知评审人 → 评审人查看需求 → 提交评审意见
→ 汇总评审结论 → 通过/驳回
```

**评审配置**：
- 评审类型：同行评审、技术评审、合规评审
- 评审人：按需求层级自动推荐（URS→产品经理+合规专员；SRS→系统架构师+合规专员；DRS→系统架构师+研发工程师）
- 评审意见字段：结论（通过/有条件通过/驳回）、意见类型（建议/问题/致命）、详细描述、关联需求片段

**验收标准**：
- Given 需求状态为"草稿" / When 创建人提交评审 / Then 系统通知所有评审人，需求状态变为"评审中"，评审截止时间默认7个工作日
- Given 评审人提交评审意见 / When 所有评审人完成评审 / Then 系统汇总评审结论：无致命问题且通过率>50%→"已通过"；否则→"已驳回"
- Given 评审结论为"有条件通过" / When 创建人确认修改意见 / Then 系统记录修改内容，状态变为"已通过"
- Given 任意评审发生 / When 评审完成 / Then 评审记录（评审人、时间、意见、结论）永久保存，不可删除

#### 7.2.4 基线快照管理（FR-0.8）

**功能描述**：对任意层级的需求集合创建基线快照，基线不可修改但可对比差异。

**验收标准**：
- Given 项目经理选择一组需求 / When 创建基线 / Then 系统生成基线快照（含所有需求的当前状态、字段值、追溯关系），基线编号格式：BL-{项目编号}-{序号}-{日期}
- Given 基线已创建 / When 用户尝试修改基线内容 / Then 系统拒绝操作并提示"基线不可修改"
- Given 基线已创建 / When 用户选择与另一基线对比 / Then 系统展示两个基线之间的需求差异（新增/删除/修改的需求列表，字段级变更详情）
- Given 项目到达DCP阶段门 / When 项目经理创建基线 / Then 系统自动命名为"DCP{N}-基线-{日期}"

#### 7.2.5 拆解工作台（FR-0.7 / FR-1.1）

**功能描述**：核心工作界面，支持需求的逐层拆解转化。

**工作台功能**：
1. **树形视图**：左侧展示当前项目的需求层级树，支持展开/折叠
2. **拆解面板**：右侧展示选中需求的详情和拆解操作区
3. **拆解操作**：选择上游需求→点击"创建下游需求"→自动继承上游关联→填写下游字段→保存
4. **覆盖率校验**：拆解完成后，系统自动校验下游验收标准是否覆盖上游验收标准
5. **看板视图**（P1增强）：拖拽卡片式拆解，支持并行拆解多条需求

**验收标准**：
- Given 用户在拆解工作台选择URS-001 / When 点击"拆解为PRS" / Then 系统创建PRS草稿，自动关联URS-001，自动继承URS-001的法规关联
- Given 拆解工作台中有3条PRS关联到URS-001 / When 用户点击"校验覆盖率" / Then 系统显示URS-001每条验收标准对应的PRS覆盖状态（已覆盖/未覆盖/部分覆盖）
- Given 拆解完成 / When 所有上游验收标准均已覆盖 / Then 上游需求状态自动更新为"已拆解"

### 7.3 追溯管理

#### 7.3.1 纵向追溯（FR-0.5）

**功能描述**：URS→PRS→SRS→DRS四层纵向追溯链路的建立、维护和查询。

**追溯链路类型**：

| 追溯方向 | 含义 | 建立时机 |
|---------|------|---------|
| 正向追溯（自上而下） | 上游需求→下游需求，回答"这条需求被哪些下游需求实现" | 拆解转化时自动建立 |
| 反向追溯（自下而上） | 下游需求→上游需求，回答"这条需求来源于哪条上游需求" | 拆解转化时自动建立 |

**验收标准**：
- Given URS-001拆解为PRS-010/PRS-011 / When 用户查看URS-001的正向追溯 / Then 系统显示PRS-010、PRS-011，以及进一步向下的SRS、DRS完整链路
- Given DRS-020-01 / When 用户查看其反向追溯 / Then 系统显示SRS-020→PRS-010→URS-001的完整上游链路
- Given 任意需求 / When 追溯链路中存在断裂（某层级无上下游关联） / Then 系统在该需求上显示"追溯断裂"警告标识

#### 7.3.2 横向追溯

**功能描述**：同一层级需求之间的关联关系管理。

| 横向关联类型 | 含义 | 示例 |
|------------|------|------|
| 依赖 | A必须在B之前完成 | SRS-020依赖SRS-010（数据格式定义） |
| 冲突 | A与B存在逻辑矛盾 | URS-001与URS-005对精度要求冲突 |
| 复用 | A与B共享同一实现 | PRS-010复用PRS-003的通信模块 |

**验收标准**：
- Given 用户创建需求A / When 设置A与需求B的横向关联 / Then 系统记录关联类型和双向关系，B的需求详情页也显示与A的关联
- Given 需求A与B存在"冲突"关联 / When 任一方发生变更 / Then 系统自动通知另一方的负责人

#### 7.3.3 追溯矩阵（FR-0.5 / FR-1.4）

**功能描述**：一键生成需求追溯矩阵，支持导出。

**矩阵格式**：

| URS-ID | URS标题 | PRS-ID | PRS标题 | SRS-ID | SRS标题 | DRS-ID | DRS标题 | 测试用例 | 状态 |
|--------|---------|--------|---------|--------|---------|--------|---------|---------|------|
| URS-001 | ... | PRS-010 | ... | SRS-020 | ... | DRS-020-01 | ... | TC-001 | ✓ |

**验收标准**：
- Given 项目中存在四层需求 / When 用户点击"生成追溯矩阵" / Then 系统在≤5秒内生成完整矩阵，包含所有纵向追溯关系
- Given 追溯矩阵已生成 / When 用户点击"导出" / Then 系统导出为Excel/PDF格式，文件名格式"追溯矩阵-{项目编号}-{日期}"
- Given 追溯矩阵中存在断裂（某层级无下游） / Then 对应单元格显示红色标记"未覆盖"

#### 7.3.4 追溯链断裂自动检测（FR-0.9）

**功能描述**：实时检测未建立追溯关系的孤立需求。

**检测规则**：

| 检测类型 | 规则 | 严重级别 |
|---------|------|---------|
| 孤立需求 | URS无下游PRS | 高 |
| 悬空需求 | PRS/SRS/DRS无上游追溯 | 高 |

| 测试缺口 | SRS/DRS无关联测试用例 | 中 |
| 法规缺口 | 需求关联的法规条款未在下游继承 | 高 |

**验收标准**：
- Given URS-002通过评审超过7天且无下游PRS / When 系统执行断裂检测 / Then URS-002被标记为"孤立需求-高"，并通知产品经理和项目经理
- Given 断裂检测发现问题 / When 用户查看仪表盘 / Then 仪表盘显示断裂统计卡片（按类型和严重级别分组），点击可跳转到对应需求列表
- Given 新的追溯关系被建立 / When 断裂问题被修复 / Then 对应标记自动消除

### 7.4 变更管理

> **设计原则**：变更管理不是需求管理的子环节，而是跨模块枢纽功能。变更影响需求、追溯、合规、风险、测试全链路。

#### 7.4.1 变更发起与分类（FR-1.7）

**变更类型**：

| 变更类型 | 触发条件 | 审批级别 |
|---------|---------|---------|
| 纠正性变更 | 需求描述错误/不完整 | 项目经理审批 |
| 适应性变更 | 法规/标准更新 | 合规专员+项目经理审批 |
| 完善性变更 | 功能优化/新需求 | 项目经理+研发总监审批 |
| 紧急变更 | 安全/合规风险 | 研发总监直接审批，事后补流程 |

**变更申请必填字段**：
- 变更原因（枚举+详细描述）
- 变更内容（修改前后对比）
- 影响范围（系统自动填充，人工确认）
- 紧急程度（普通/紧急）
- 期望完成日期

**验收标准**：
- Given 基线化后的需求 / When 用户点击"发起变更" / Then 系统弹出变更申请表单，自动填充需求当前内容
- Given 变更申请已提交 / When 审批人查看 / Then 可看到修改前后对比（diff视图）、自动影响评估报告、审批操作

#### 7.4.2 变更影响自动评估（FR-1.3）

**功能描述**：变更提交时自动生成影响范围报告。

**影响评估维度**：

| 维度 | 评估内容 | 自动/手动 |
|------|---------|----------|
| 下游需求 | 受影响的PRS/SRS/DRS列表 | 自动 |
| 测试用例 | 受影响的测试用例列表 | 自动 |
| 法规影响 | 受影响的法规条款合规性 | 自动 |
| 横向关联 | 存在依赖/冲突关系的同层需求 | 自动 |
| 工期影响 | 预估影响的工作量（人天） | 手动填写 |
| 风险影响 | 变更引入的新风险 | 手动填写 |

**验收标准**：
- Given URS-001发起变更 / When 系统执行影响评估 / Then 自动列出所有下游PRS/SRS/DRS、关联测试用例、法规条款，计算影响范围（受影响项数/总关联项数×100%）
- Given 影响评估完成 / When 项目经理查看报告 / Then 报告包含：受影响需求数、受影响测试用例数、法规影响标记、预估影响范围百分比

#### 7.4.3 下游自动标记suspect（FR-0.10）

**功能描述**：上游需求变更后，所有下游关联项自动标记为"suspect"（源自Jama Connect最佳实践）。

**验收标准**：
- Given URS-001变更已批准 / When 系统处理变更 / Then 所有下游PRS/SRS/DRS自动标记为"suspect"，状态不变但显示黄色警告标识
- Given 下游需求被标记为"suspect" / When 负责人确认无需修改 / Then 手动清除"suspect"标记，系统记录确认人和时间
- Given 下游需求被标记为"suspect" / When 负责人修改需求 / Then 修改保存后"suspect"标记自动清除，系统记录修改内容

#### 7.4.4 变更审批流（FR-1.7）

**审批流程**：

```
变更申请 → 影响评估 → 审批 → 执行 → 验证 → 关闭
```

| 阶段 | 执行者 | 产出 |
|------|--------|------|
| 变更申请 | 需求负责人 | 变更申请单 |
| 影响评估 | 系统+项目经理 | 影响评估报告 |
| 审批 | 按变更类型确定审批人 | 审批结论 |
| 执行 | 需求负责人 | 修改后需求 |
| 验证 | 测试工程师+质量工程师 | 验证结论 |
| 关闭 | 项目经理 | 变更记录归档 |

**验收标准**：
- Given 变更审批通过 / When 需求负责人执行修改 / Then 系统记录修改diff，更新追溯关系，通知所有受影响项的负责人
- Given 变更执行完成 / When 测试工程师验证 / Then 验证结论记录在变更单中，所有"suspect"标记根据验证结果清除或保留
- Given 变更全流程完成 / When 项目经理关闭变更 / Then 变更单归档，审计日志完整记录全过程

#### 7.4.5 追溯链自动更新

**功能描述**：变更执行后，自动更新追溯关系。

**验收标准**：
- Given 变更删除了某条下游需求 / When 变更执行 / Then 上游需求的正向追溯自动移除该关联，追溯断裂检测重新执行
- Given 变更新增了某条下游需求 / When 变更执行 / Then 新追溯关系自动建立，法规关联自动继承
- Given 变更修改了需求层级 / When 变更执行 / Then 追溯矩阵自动更新，相关基线对比报告自动刷新

### 7.5 合规管理

#### 7.5.1 电子签名（FR-0.2）

**功能描述**：符合21 CFR Part 11电子签名要求。

**签名要求**：

| 操作类型 | 签名含义 | 签名方式 |
|---------|---------|---------|
| 需求创建 | "我是本文档的创建者" | 用户名+密码+签名含义 |
| 需求审批 | "我批准此需求" | 用户名+密码+签名含义 |

| 变更审批 | "我批准此变更" | 用户名+密码+签名含义 |
| 基线确认 | "我确认此基线" | 用户名+密码+签名含义 |

**验收标准**：
- Given 用户执行需要电子签名的操作 / When 系统弹出签名对话框 / Then 用户输入密码和签名含义，系统验证身份后记录签名

**功能描述**：符合21 CFR Part 11审计追踪要求。

**审计日志记录范围**：

| 事件类型 | 记录内容 |
|---------|---------|
| 创建 | 操作人、时间、创建的内容摘要 |
| 修改 | 操作人、时间、修改前后值（字段级） |
| 删除 | 操作人、时间、删除的内容摘要 |
| 状态变更 | 操作人、时间、原状态→新状态、变更原因 |
| 签名 | 签名人、签名时间、签名含义、文档哈希 |
| 登录/登出 | 用户、时间、IP地址 |
| 权限变更 | 操作人、时间、变更内容 |

**验收标准**：
- Given 任意系统操作发生 / When 操作完成 / Then 审计日志自动记录，延迟≤1秒
- Given 审计日志已记录 / When 任何用户尝试删除或修改日志 / Then 系统拒绝操作
- Given 合规专员查询审计日志 / When 按时间范围/操作人/事件类型筛选 / Then 系统返回匹配日志，支持导出

### 7.7 项目管理（第二阶段交付）

> **设计原则**：项目管理与需求管理原生集成，先"需求-任务联动"，再"完整项目管理"（INS-5）。
>
> **✅Q5已确认**：中旗目前无项目管理系统，本模块为**全新建设**，无需考虑与禅道等工具的数据迁移或并存问题。

#### 7.7.1 需求→任务转化（FR-1.10 / FR-2.6）

**功能描述**：SRS/DRS可拆解为任务，实现需求到执行层的转化。

**转化路径**：

```
SRS/DRS → 1:N 拆解 → 任务
```

**任务字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| 任务ID | 自动生成 | TSK-{项目编号}-{序号} |
| 任务标题 | 文本 | ≤50字 |
| 详细描述 | 富文本 | 任务详细说明 |
| 上游需求 | 关联 | 关联的SRS/DRS-ID |
| 负责人 | 关联 | 研发工程师 |
| 优先级 | 枚举 | 紧急/高/中/低 |
| 预估工时 | 数值 | 小时 |
| 实际工时 | 数值 | 小时（填报） |
| 开始日期 | 日期 | 计划开始 |
| 截止日期 | 日期 | 计划完成 |
| 依赖任务 | 多选关联 | 前置任务ID列表 |
| 状态 | 枚举 | 待开始/进行中/已完成/已阻塞 |

**任务状态与需求状态联动规则**：

| 任务状态变化 | 需求状态影响 |
|------------|------------|
| 所有任务"已完成" | DRS状态自动变为"已实现" |
| 任一任务"已阻塞"超过3天 | 通知项目经理，DRS标记为"风险" |
| 任务优先级为"紧急" | 对应需求优先级建议提升 |
| 任务实际工时>预估工时×150% | 通知项目经理，标记为"工时超支" |

**验收标准**：
- Given SRS-020已拆解为DRS / When 项目经理将DRS-020-01拆解为3个任务 / Then 每个任务自动关联DRS-020-01，继承其法规关联和验收标准
- Given DRS-020-01的所有任务标记为"已完成" / When 最后一个任务完成 / Then DRS-020-01状态自动更新为"已实现"，通知测试工程师
- Given 任务TSK-001阻塞超过3天 / When 系统检测 / Then 自动通知项目经理，DRS标记"风险"

#### 7.7.2 里程碑管理（FR-2.6）

**功能描述**：项目里程碑与DCP阶段门映射。

**里程碑定义**：

| 里程碑 | 对应DCP | 达成条件 |
|--------|---------|---------|
| M1: 概念评审通过 | DCP1 | 所有URS"已通过"，法规关联完整 |
| M2: 计划评审通过 | DCP2 | 所有PRS"已通过"，追溯完整率≥90% |
| M3: 开发完成 | DCP3 | 所有DRS"已实现"，研发自验完成 |
| M4: 验证完成 | DCP4 | 所有DRS"已验证"，测试覆盖率100% |
| M5: 发布就绪 | DCP5 | 追溯闭环率≥95%，合规证据包可生成 |

**验收标准**：
- Given 项目经理创建项目 / When 系统初始化 / Then 自动创建M1-M5五个里程碑，每个关联对应的DCP阶段门
- Given 里程碑达成条件全部满足 / When 系统检测 / Then 自动标记里程碑为"可达成"，通知研发总监
- Given 里程碑计划日期到达但条件未满足 / When 系统检测 / Then 标记为"延期"，通知项目经理和研发总监

## 8. 指标体系与度量

### 8.1 北极星指标

| 指标名 | 定义 | 公式 | 采集方式 | 预警阈值 | 负责人 |
|--------|------|------|---------|---------|--------|
| 需求全链路闭环率 | 从URS到DRS+测试验证全部闭环的需求占比 | 全链路闭环需求数 / 总需求数 × 100% | 系统自动计算 | < 85% | 项目经理 |

### 8.2 一级指标

| 指标名 | 定义 | 公式 | 采集方式 | 预警阈值 | 负责人 |
|--------|------|------|---------|---------|--------|
| 合规达标率 | 审计日志完整性、电子签名覆盖率、需求合规率的综合指标 | (审计日志完整率 + 电子签名覆盖率 + 需求合规率) / 3 | 系统自动计算 | < 90% | 合规专员 |
| 追溯完整率 | 需求纵向追溯链路的完整性 | (已建立完整追溯链的需求数 / 总需求数) × 100% | 系统自动计算 | < 90% | 系统架构师 |
| 需求交付效率 | 需求从创建到验证通过的平均周期 | 需求平均交付周期（天） | 系统自动计算 | > 30天 | 项目经理 |
| 风险可控率 | 高风险需求占比处于可控范围 | 1 - (高风险需求数 / 总需求数) | 系统自动计算 | 高风险占比 > 20% | 质量工程师 |

### 8.3 二级指标

| 指标名 | 定义 | 公式 | 采集方式 | 预警阈值 | 负责人 | 所属一级指标 |
|--------|------|------|---------|---------|--------|------------|
| 审计日志完整性 | 关键操作有审计日志的比例 | 有日志记录的关键操作数 / 关键操作总数 × 100% | 系统自动计算 | < 100% | 合规专员 | 合规达标率 |
| 电子签名覆盖率 | 需要签名的操作已签名的比例 | 已签名操作数 / 需签名操作总数 × 100% | 系统自动计算 | < 100% | 合规专员 | 合规达标率 |
| 需求合规率 | 法规关联完整的需求占比 | 法规关联完整需求数 / 总需求数 × 100% | 系统自动计算 | < 95% | 合规专员 | 合规达标率 |
| 不合规项整改率 | 不合规项已整改的比例 | 已整改不合规项数 / 不合规项总数 × 100% | 手动标记 | < 90% | 质量工程师 | 合规达标率 |
| 需求-设计覆盖率 | DRS覆盖SRS的比例 | 已覆盖SRS数 / 总SRS数 × 100% | 系统自动计算 | < 95% | 系统架构师 | 追溯完整率 |
| 需求-测试用例覆盖率 | 测试用例覆盖SRS/DRS的比例 | 已关联测试用例的SRS/DRS数 / 总SRS/DRS数 × 100% | 系统自动计算 | < 100% | 测试工程师 | 追溯完整率 |
| 追溯闭环率 | 有完整上下游链路的需求占比 | 有完整链路的需求数 / 总需求数 × 100% | 系统自动计算 | < 85% | 系统架构师 | 追溯完整率 |
| 变更追溯覆盖率 | 变更后追溯链路更新完整的比例 | 追溯更新完整的变更数 / 总变更数 × 100% | 系统自动计算 | < 100% | 项目经理 | 追溯完整率 |
| 需求平均交付周期 | 需求从创建到验证通过的平均天数 | Σ(验证通过时间-创建时间) / 已验证需求数 | 系统自动计算 | > 30天 | 项目经理 | 需求交付效率 |
| 需求评审通过率 | 一次评审通过的需求占比 | 一次通过需求数 / 总评审需求数 × 100% | 系统自动计算 | < 70% | 产品经理 | 需求交付效率 |
| 变更平均审批周期 | 变更从申请到审批完成的平均天数 | Σ(审批完成时间-申请时间) / 总变更数 | 系统自动计算 | > 5天 | 项目经理 | 需求交付效率 |
| 需求一次通过率 | 评审一次通过的需求占比（含条件通过） | (一次通过+有条件通过) / 总评审数 × 100% | 系统自动计算 | < 60% | 产品经理 | 需求交付效率 |
| 高风险需求占比 | RPN≥120的需求占比 | 高风险需求数 / 总需求数 × 100% | 系统自动计算 | > 20% | 质量工程师 | 风险可控率 |
| 未闭环风险数 | 状态非"关闭"的风险记录数 | COUNT(风险状态≠关闭) | 系统自动计算 | > 5条 | 质量工程师 | 风险可控率 |
| 风险处理及时率 | 在规定时间内完成风险处理的比例 | 及时处理风险数 / 总风险数 × 100% | 系统自动计算 | < 80% | 质量工程师 | 风险可控率 |
| 风险复发率 | 已关闭风险再次发生的比例 | 复发风险数 / 已关闭风险数 × 100% | 手动标记 | > 10% | 质量工程师 | 风险可控率 |
| 需求返工率 | 需要修改后重新评审的需求占比 | 返工需求数 / 总评审需求数 × 100% | 系统自动计算 | > 30% | 产品经理 | 需求交付效率 |
| 需求描述完整性 | 必填字段全部填写的需求占比 | 字段完整需求数 / 总需求数 × 100% | 系统自动计算 | < 100% | 产品经理 | 需求交付效率 |

### 8.4 项目管理指标（第二阶段）

| 指标名 | 定义 | 公式 | 采集方式 | 预警阈值 | 负责人 |
