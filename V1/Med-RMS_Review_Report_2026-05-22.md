## 二、PRD 评审意见（产品经理 · 许清楚）

### 🔴 严重问题（3项，必须开发前修订）

#### P-RED-01：用户故事缺乏可测试的验收标准

**问题描述**：8个用户故事缺少清晰、可量化、可测试的验收标准。没有验收标准的用户故事无法被开发和测试团队有效执行。例如"作为RA专员，我想生成追溯矩阵报告"——验收标准应明确：报告包含哪些层级？覆盖率如何计算？格式是什么？导出格式有哪些？

**影响**：开发阶段对"完成"定义不一致，测试无法覆盖异常场景，Sprint验收时无客观标准。

**改进建议**：
1. 每个用户故事必须配套 Given-When-Then 格式验收标准
2. 验收标准至少包含：① 功能验收条件 ② 合规验收条件（对应法规条款）③ 性能/数据量验收条件
3. 建议以表格形式逐一列出，P0用户故事必须完成

---

#### P-RED-02：北极星指标"合规零缺陷"定义模糊，不可度量

**问题描述**：产品目标 G1 为"合规零缺陷"，但"零缺陷"在软件工程中既不现实也不可操作。FDA 和 ISO 认证审计从不要求"零缺陷"，而是要求"缺陷有受控的处理过程"。将"零缺陷"作为北极星指标会导致：① 团队为追求指标而隐藏缺陷 ② 无法衡量改进 ③ 与 IEC 62304 Clause 9 的"问题解决过程"理念矛盾——该条款承认问题会发生，关键是有系统化的处理流程。

**影响**：指标不可度量就无法驱动决策，还可能误导团队行为。

**改进建议**：将 G1 改为可度量指标，例如：
- "审计发现项闭环率 ≥ 95%"（闭环 = 有根本原因分析 + 纠正措施 + 验证确认）
- "合规性检查通过率 ≥ 98%"（定义明确的合规检查清单）
- "关键/重大缺陷发现至闭环的周期 ≤ 30天"

---

#### P-RED-03：IEC 62304 覆盖存在重大缺口

**问题描述**：基于对 IEC 62304 全部条款的逐项审查，以下关键要求在 PRD 中存在明显缺口：

| IEC 62304 条款 | 要求 | PRD 覆盖情况 |
|---|---|---|
| Clause 5.1 软件开发策划 | 软件开发计划（含可追溯性策略、配置管理、变更控制） | 部分覆盖（变更管理有，开发策划未提） |
| Clause 5.2 软件需求分析 | SRS + 需求可追溯性矩阵 | 部分覆盖（需求管理有，可追溯性矩阵未明确） |
| Clause 5.3 软件架构设计 | 架构文档 + **SOUP 组件记录** | ❌ 缺失（SOUP 管理未提） |
| Clause 5.7 软件系统测试 | 系统测试与需求追溯 | 部分覆盖（追溯管理有，测试关联未明确） |
| Clause 6 软件维护过程 | 维护计划 + 问题报告 + 变更请求 + 通知义务 | 部分覆盖（变更管理有，问题报告和通知义务未提） |
| Clause 7 软件风险管理 | 危险源识别→风险控制→验证→**SOUP 风险** | 部分覆盖（风险管理模块有，SOUP 风险未提） |
| Clause 8 配置管理 | 配置标识 + 变更控制 + **基线管理** + SOUP 记录 | 部分覆盖（变更控制有，基线和 SOUP 未提） |
| Clause 9 问题解决 | 问题报告 + 根因分析 + 趋势分析 | ❌ 缺失（问题报告管理未独立） |

**影响**：对中旗的心电图机等 Class B/C 医疗器械软件，IEC 62304 是 NMPA 注册审评的必查项。缺失此映射可能导致合规审计不符合项。

**改进建议**：
1. 在 PRD 增加"IEC 62304 生命周期过程映射表"，将 URS/PRS/SRS/DRS 四层模型与 IEC 62304 Ch.5-9 各条款明确对应
2. 补充 SOUP 管理（Clause 5.3/7/8）
3. 补充问题报告管理（Clause 9）
4. 在需求字段中增加"软件安全分类"（Class A/B/C）
5. 在合规模块中增加 IEC 62304 合规检查清单

### 🟠 重要问题（10项，建议修订）

#### P-ORG-01：缺少 SOUP（未知来源软件）管理场景

**问题描述**：IEC 62304 Clause 5.3.3/7.1.2/8.1.2 均对 SOUP 管理提出明确要求——须记录 SOUP 名称/供应商/版本号、审查其异常列表/已知问题、评估其对安全的潜在影响。PRD 的需求管理和风险管理模块均未体现。对于中旗这类嵌入式医疗器械企业，SOUP（含第三方库、开源组件、遗留代码）是高频痛点。

**改进建议**：在需求管理或风险管理模块中增加 SOUP 登记与追踪子功能，包括：SOUP 组件清单登记、版本与供应商记录、已知异常列表审查记录、安全影响评估关联。

---

#### P-ORG-02：缺少软件安全分类（Class A/B/C）管理场景

**问题描述**：IEC 62304 核心概念——基于潜在伤害严重度将软件分为 Class A/B/C，不同等级对应不同的文档和活动要求。PRD 的四层需求架构未体现安全分类维度，无法按安全等级差异化管控。

**改进建议**：在需求条目中增加「软件安全分类」属性（A/B/C），并根据分类自动匹配 IEC 62304 规定的必需文档和活动清单。

---

#### P-ORG-03：关键角色用户故事覆盖严重不足

**问题描述**：8类用户角色仅有8个用户故事，平均每类角色1个故事。尤其缺少：QA工程师审核测试追溯完整性、RA专员生成合规申报材料、项目经理监控变更影响范围、开发者查看关联需求。

**改进建议**：每个角色至少补充2-3个核心用户故事，总量应达到16-24个。重点关注合规审计、追溯验证、变更影响分析三大高频场景。

---

#### P-ORG-04：P0 工期（80人天）可能被低估

**问题描述**：仅审计追踪一项就需要10-15人天，电子签名实现约10人天，四层需求层级转化逻辑约10人天，追溯矩阵生成约8人天。80人天覆盖 P0 全部核心功能偏紧。

**改进建议**：对 P0 逐项拆估，特别关注合规功能开发量（通常是普通功能的1.5-2倍）。如果确认80人天不足，应将部分 P0 降为 P1 并调整 MVP 范围，而非压缩工期。

---

#### P-ORG-05："追溯全链路≥85%"的阈值缺乏依据

**问题描述**：未说明覆盖率如何计算（按需求条目数？按层级对数？），85%的依据是什么，剩余15%未追溯需求如何处理。IEC 62304 实际要求"所有需求均可追溯"（Clause 5.2.2），并不接受部分追溯。

**改进建议**：
- 重新定义覆盖率计算方式：已建立完整追溯链的需求条目数 / 总需求条目数 × 100%
- 将目标设为分阶段递进：MVP ≥85%，V1.0 ≥95%，V2.0 ≥99%
- 对未覆盖的需求必须有"未追溯原因"记录和风险评估

---

#### P-ORG-06：缺少过程类指标

**问题描述**：当前指标体系以结果指标为主，缺少过程指标。医疗器械合规强调过程合规（IEC 62304 本身就是过程标准）。

**改进建议**：增加关键过程指标：
- 需求评审周期（提交→审批完成）：≤5个工作日
- 变更影响分析完成率：100%（每个变更必须完成影响分析后方可审批）
- 追溯矩阵更新及时率：需求变更后24h内更新关联追溯

---

#### P-ORG-07：竞品分析缺少国产竞品和本地化优势分析

**问题描述**：PRD 列出的竞品（IBM DOORS、Jama Connect、Polarion MedPack）均为海外产品。对中旗这样的国产医疗器械企业，最真实的竞品是"Excel/Word + 人工流程"和国产 ALM 工具（禅道、PingCode等），未分析这些替代品意味着可能高估了差异化优势。

**改进建议**：
- 补充国产 ALM 工具竞品分析
- 补充"Excel/Word 人工流程"作为竞品——最真实的替代方案
- 明确 Med-RMS 核心差异化："专为医疗器械合规设计的四层需求架构 + 内置法规条款映射"

---

#### P-ORG-08：工期与资源估算不匹配，缺少 CSV 工期规划

**问题描述**：16个月总工期中 P0 仅约4人月，P1/P2 占比过大，且团队规模与人天估算不匹配。更关键的是，医疗器械合规系统自身也需要经过计算机系统验证（CSV），21 CFR Part 11 11.10(a) 明确要求，CSV 工作量通常占开发总工期的20-30%，但 PRD 未单独规划。

**改进建议**：
- 明确团队规模和角色配置
- 重新校准人天估算（合规功能开发量 ×1.5-2）
- 在交付计划中单独列出 CSV 工期（建议2-3个月），与开发并行启动
- 预留数据迁移1-2个月、用户培训2-4周、并行运行期1-2个月

---

#### P-ORG-09：Non-goal"不包含硬件需求管理"可能过于绝对

**问题描述**：URS 层面需求通常同时包含硬件和软件需求，如果完全不管理硬件需求，则 URS→PRS 转化中硬件需求成为"孤岛"，破坏追溯完整性。IEC 62304 Clause 5.2.2 要求的可追溯性是端到端的。

**改进建议**：
- 方案A（推荐）：在 URS 层支持记录硬件需求条目（轻量级，仅编号、描述、状态），但不提供深度管理
- 方案B：至少支持 URS 条目标注"硬件/软件/软硬件"分类，并提供外部链接字段

---

#### P-ORG-10：21 CFR Part 11 覆盖不完整

**问题描述**：逐项审查发现多项缺口：

| Part 11 要求 | PRD 覆盖情况 |
|---|---|
| 11.10(a) 系统验证 | ❌ 未提及系统自身的 CSV 计划 |
| 11.10(e) 审计追踪 | 部分覆盖（需确认旧值/新值/操作人/时间/原因全记录、防篡改） |
| 11.10(f) 操作序列检查 | ❌ 缺失（如审批前必须有评审记录） |
| 11.50 签署显现 | 需确认 |
| 11.70 签名与记录链接 | 需确认 |
| 11.100 签名唯一性 | 需确认 |
| 11.200 双组件签名 | 需确认 |
| 11.300 令牌/密码控制 | 部分覆盖（密码策略需确认） |

**改进建议**：
- 增加 21 CFR Part 11 合规检查清单
- 增加操作序列强制检查功能
- 明确审计追踪详细字段要求（旧值/新值/操作人/时间/原因，防篡改，不可删除）

## 三、架构文档评审意见（架构师 · 高见远）

### 🔴 严重问题（3项，必须开发前修订）

#### A-RED-01：缺少 21 CFR Part 11 电子签名方案

**问题描述**：当前安全架构仅提及"RBAC + JWT + HTTPS + API Key + 审计日志只读约束"，**完全未涉及 21 CFR Part 11 电子签名要求**。JWT 只能证明"谁发的请求"，不等于"谁做了签名"。医疗器械软件中，需求的审批、变更的确认、文档的签发等关键操作必须使用符合 21 CFR Part 11 的电子签名。

21 CFR Part 11 核心要求与当前方案差距：

| Part 11 要求 | 当前方案 |
|---|---|
| 签名与签名者不可分离关联 | ❌ 无 |
| 签名不可被伪造 | ❌ 无签名值、无非对称加密 |
| 签名含义明确（审批/确认/拒绝等） | ❌ 无 |
| 签名时间由可信时间源提供 | ❌ 无 |
| 签名后文档不可篡改 | ❌ 无签名-文档绑定 |
| 签名者身份经过验证（至少双因素） | ❌ 仅 JWT 单因素 |

**影响**：这是医疗器械软件的硬性合规红线，不解决则系统无法通过审计。

**改进建议**：
1. 引入独立电子签名限界上下文
2. 签名时要求二次认证（密码确认或动态口令）
3. 签名值 = 非对称加密(签名者ID + 签名含义 + 签名时间 + 被签名文档哈希)
4. 电子签名数据模型：signer_id, signer_meaning(APPROVE/REJECT/REVIEW/CONFIRM), signed_at(可信时间源), signature_value, entity_type, entity_id, entity_hash, reason

---

#### A-RED-02：单表继承（STI）四层需求存在严重查询和性能风险

**问题描述**：四层需求使用单表继承，将所有层级存储在同一张表中，存在：
- **查询复杂度爆炸**：需求追溯矩阵查询需要多层级自连接（4层），性能随数据量急剧恶化
- **数据完整性约束困难**：不同层级需求字段约束不同，单表难以用 NOT NULL 约束
- **索引策略复杂**：跨层级查询需要多跳索引扫描
- **单表膨胀**：中旗多产品线，单表可能达10万+行，JSONB extended_fields 进一步增加行宽
- **状态机差异化实现复杂**：四层需求状态机各有差异，单表继承需应用层类型判断分发

**影响**：P0 阶段数据量小问题不明显，P1/P2 阶段查询性能和维护成本将显著上升。

**改进建议（推荐）**：闭包表（Closure Table）+ 分层子表
- 主表 requirement(id, type, project_id, ...) 存公共字段
- 子表 user_requirement, system_requirement 等存层级特有字段
- 闭包表 requirement_ancestor(descendant_id, ancestor_id, depth) — 追溯矩阵查询 O(1) 级
- 最小改动方案：保留单表但增加物化路径 path 列（如 "/1/5/12/20/"）

---

#### A-RED-03：审计日志缺少哈希链和数据库层强制保护

**问题描述**：追加只写是正确方向，但不够：
- 缺少数据库层面强制约束，有数据库写权限的人可篡改日志
- 缺少审计日志哈希链，21 CFR Part 11 要求"cannot be altered in a way that obscures prior values"
- 缺少归档策略

**影响**：审计日志若可被篡改，则整个合规体系的基础被动摇。

**改进建议**：
1. 审计日志增加哈希链：每条记录包含前一条记录的 SHA-256 哈希，形成链式结构
2. 数据库触发器阻止 UPDATE 和 DELETE 操作
3. 定期校验哈希链完整性
4. 审计日志至少保留10年（ISO 13485 要求）

---

### 🟠 重要问题（5项，建议修订）

#### A-ORG-01：JWT 方案缺少安全细节，且无法满足电子签名要求

**问题描述**：JWT 只能用于 API 访问认证，不等于电子签名认证。还缺少：JWT 撤销机制（Token 黑名单+Redis）、刷新策略（Access 15min + Refresh 24h）、JWT Payload 仅含 user_id（角色实时查询避免不同步）、泛微 OA 对接用 OAuth2 授权码模式。

**改进建议**：
1. JWT 仅用于 API 访问认证，不用于电子签名认证
2. 电子签名操作使用独立的签名认证流程：弹出对话框→输入密码→后端验证→记录签名
3. JWT 黑名单持久化到数据库（非仅 Redis），确保审计可追溯

---

#### A-ORG-02：跨模块通信事件驱动方案可靠性不足

**问题描述**：未说明事件机制、丢失处理、顺序保证、失败重试。Spring ApplicationEvent 进程内事件在应用崩溃时丢失。

**改进建议**：采用事务性发件箱模式（Transactional Outbox + Debezium CDC），无需引入外部消息队列，适合当前规模，保证业务操作和事件发布原子性，天然支持未来微服务拆分。

---

#### A-ORG-03：遗漏5个关键 ADR

**问题描述**：当前7个 ADR 缺少以下关键决策：
- ADR-008: 电子签名方案（🔴最高优先级）
- ADR-009: 数据库设计策略（层级数据+JSONB选型理由）
- ADR-010: 事务性发件箱 vs 消息队列
- ADR-011: 泛微OA集成方案
- ADR-012: 数据备份与灾难恢复策略

**改进建议**：Sprint0 即补充这5个 ADR，尤其是 ADR-008 电子签名方案。

---

#### A-ORG-04：遗漏5类关键 API

**问题描述**：当前 API 设计缺少：
- 电子签名 API（签名提交/验证/重签）
- 审计日志 API（查询/完整性校验/导出）
- 追溯矩阵 API（追溯查询/缺口分析）
- 变更控制 API（变更请求/影响分析/审批）
- 合规报告 API（追溯性报告/审计追踪报告）

**改进建议**：在 API 设计章节补充这5类 API 的端点定义、请求/响应 Schema。

---

#### A-ORG-05：PRD 对齐度不足——合规相关功能遗漏

**问题描述**：
- 电子签名 ❌ 未覆盖
- 合规报告 ❌ 未覆盖
- 追溯缺口分析 🟡 部分覆盖
- 文档管理方案未明确
- IEC 62304 软件生命周期过程追溯缺失

**改进建议**：与产品经理协同，以 PRD 评审中的 IEC 62304 缺口清单为输入，逐项补充架构设计。

## 六、评审总结

### 亮点

1. **PRD 结构完整**：从产品目标→用户故事→需求池→功能模块→指标体系→交付计划，逻辑链路清晰，MVP 优先+分阶段迭代的思路正确
2. **架构决策有据**：7个 ADR 都有背景→决策→后果的完整推理，避免"因为大家都在用"式选型
3. **模块化单体选择正确**：<200用户+小团队场景下，模块化单体是最务实的选择
4. **变更管理作为跨模块枢纽**：suspect 标记+事件驱动传播，设计理念先进
5. **合规意识强**：PRD 中 21 CFR Part 11 的电子签名+审计追踪方案体现了行业理解

### 主要风险

1. **🔴 合规红线风险**：电子签名零方案、审计日志无防篡改、IEC 62304 5大条款空白——这三个问题任何一个不解决，系统都无法通过医疗器械合规审计
2. **🔴 数据模型风险**：单表继承方案如不改，P1/P2 阶段性能和维护成本将显著增加
3. **🟠 工期风险**：P0 估算偏乐观，加上严重问题需要修订和 CSV 工期未规划，16个月总工期可能不足
4. **🟠 指标风险**：北极星指标不可度量、追溯覆盖率阈值缺依据、缺过程类指标
5. **🟠 PRD-架构对齐风险**：15项检查中8项未对齐，尤其是合规相关功能

### 下一步行动

1. **本周内**：解决6项🔴严重问题，更新 PRD 和架构文档
2. **Sprint0 期间**：解决15项🟠重要问题，确认泛微OA版本和SSO协议
3. **Sprint1 启动条件**：🔴问题全部关闭，🟠问题至少关闭10/15

### 最关键的三个改进方向

1. **以 IEC 62304 条款为骨架重构需求架构**：当前 PRD 的需求管理、追溯管理、变更管理模块基本方向正确，但缺少 SOUP 管理、安全分类、问题报告、基线管理等 IEC 62304 明确要求的核心场景。建议以 IEC 62304 Clause 5-9 为检查清单，逐条确认覆盖度。

2. **补充 21 CFR Part 11 电子签名完整方案**：这是架构文档最大的空白。电子签名不是"加个密码框"，而是一套完整的签名-验证-防篡改-审计体系。必须在开发前设计完成。

3. **重新定义指标体系和工期规划**：将"合规零缺陷"改为可度量、可操作的过程指标和结果指标组合；补充 CSV 工期（2-3个月）；数据迁移和用户培训纳入计划。
